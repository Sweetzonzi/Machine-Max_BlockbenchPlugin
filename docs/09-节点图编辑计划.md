# 09 — 节点蓝图编辑器计划

> 创建时间：2026-05-25

---

## 1. 动机与定位

### 1.1 为什么需要节点编辑器

当前 MachineMax Blockbench 插件已实现的功能覆盖了**字段级配置**（填写表单，选择下拉），但 UGC 作者面临以下瓶颈：

| 痛点 | 表单方式的局限 | 节点图的价值 |
|------|-------------|-----------|
| **信号路由看不见** | `signal_targets` / `control_outputs` 是嵌套的键值对，一眼看不出"谁发、谁收、经过哪" | 连线的有向边清晰展示端到端路由 |
| **子系统间耦合不直观** | 发动机 → 变速箱 → 传动系统 → 轮胎，这条链分散在 4 个子系统面板里 | 节点图中一目了然 |
| **状态机无法手写** | JSON 手写状态机极其容易出错（状态、转换、条件、动画） | 节点编辑器天然适配 FSM 的"状态节点 + 条件边"范式 |
| **修改成本高** | 改一个信号目标要：切换面板 → 找到子系统 → 展开信号区域 → 查找目标 → 修改 | 在节点图中直接拖拽连线即可 |

### 1.2 定位：不只是"信号拓扑图"，而是通用蓝图编辑器

本阶段的目标从 [05-开发计划 §6](./05-开发计划.md#6-阶段四信号拓扑预览与蓝图phase-2--3) 中的"Phase 3：交互式蓝图编辑器" 扩展为**通用节点蓝图编辑框架**：

```
原定位（Phase 3）：只读 SVG 拓扑图 + 拖拽连线编辑【仅信号路由】

新定位（本计划）：
  ├── 图类型 A：信号与动力路由图  ← 信号拓扑预览的自然升级
  ├── 图类型 B：状态机编辑器      ← 新需求：动画/行为状态机
  └── 图类型 C（预留）：数值计算流  ← 如传动比、坐标变换等计算式
```

## 2. 技术选型：LiteGraph.js

### 2.1 选型理由回顾

之前对 **LiteGraph.js**、Rete.js v1、Vue Flow、Butterfly、Node-RED 的完整调研结论：

| 库 | 与 Vue 2 兼容 | 渲染 | 编辑功能 | 推荐 |
|----|-------------|------|---------|------|
| **LiteGraph.js** | ✅ 独立 Canvas | Canvas (~150KB) | 完整 | ⭐ 强烈推荐 |
| Rete.js v1 | ✅ Vue 2 | DOM | 完整 | 可选（生态已停滞） |
| Vue Flow | ❌ Vue 3 only | DOM | 完整 | 不可行 |
| Butterfly | ❌ React 绑定 | SVG | 部分 | 不推荐 |
| Node-RED | ❌ jQuery 冲突 | DOM | 完整 | 不推荐 |

**关键优势**：

1. **纯 JS，零框架依赖** — 直接 `require()`，esbuild 自然打包
2. **Canvas 渲染隔离** — 与 Vue 2 虚拟 DOM 无冲突
3. **开箱即用的编辑能力** — 拖拽、连线、搜索（Ctrl+F）、撤销、框选、迷你地图
4. **体积可控** — minified ~150KB
5. **有成功先例** — AUTOMATIC1111/stable-diffusion-webui 的 ComfyUI 风格节点编辑器基于 LiteGraph

### 2.3 为什么不用现有 SVG 方案继续增强

| 对比维度 | 现有 SVG (SignalFlowPanel) | LiteGraph.js |
|---------|--------------------------|-------------|
| 交互开发量 | 拖拽连线、hit-testing、框选均需从零实现（预估 1500+ 行） | 开箱即用 |
| 撤销/重做 | 需自建命令栈 | 内置 `graph.undo()`/`redo()` |
| 节点搜索 | 需自建搜索框 + 过滤逻辑 | 内置 `showSearchBox()` |
| 迷你地图 | 需自建 | 内置 |
| 节点拖拽 | 需自建（含碰撞检测） | 内置 |
| 长期维护 | 自己维护交互代码 | 社区维护核心库 |

现有 **SignalFlowPanel.vue.js**（579 行）和 **signal_graph.js**（445 行）的资源将**部分复用**：

- `signal_graph.js` 的 `extractSignalGraph()` 逻辑 → 改造成 `toGraph()` 转换器
- `signal_graph.js` 的 `forceLayout()` → 替换为 LiteGraph 内置布局（或保留作为备选 `arrange()`）
- `SignalFlowPanel.vue.js` → 整体替换为 `NodeGraphPanel.vue.js`

## 3. 节点类型系统设计

### 3.1 节点分类

```
节点基类：MmBaseNode (extends LGraphNode)
├── 信号路由图节点 (SignalRoutingNode)
│   ├── SubsystemNode     — 子系统节点（按类型着色）
│   ├── ConnectorNode     — 连接点节点（绿色）
│   ├── InteractBoxNode   — 交互区节点（金色）
│   └── SpecialTargetNode — 特殊目标：subpart / vehicle（灰色）
│
├── 状态机节点 (StateMachineNode)
│   ├── StateNode         — 状态节点（椭圆/圆角矩）
│   ├── TransitionEdge    — 转换边（带条件标签）
│   └── EntryPointNode    — 入口节点
│
└── 计算流节点（预留）
    ├── InputNode         — 输入参数
    ├── ComputeNode       — 运算操作（加减乘除、三角函数等）
    └── OutputNode        — 输出结果
```

### 3.2 信号路由图节点规格

#### SubsystemNode（子系统节点）

```
┌──────────────────────────────┐
│  🔧 发动机                   │ ← 标题栏（类型图标 + displayName）
│  (engine)                    │ ← 副标题（类型 id）
│──────────────────────────────│
│  ▶ speed_outputs             │ ← 输出端口（蓝色）
│  ▶ power_output              │ ← 输出端口（橙色）
│                              │
│  ◀ control_channels          │ ← 输入端口（灰色）
└──────────────────────────────┘
```

端口定义从 `subsystem_types.js` 注册表的 `signalOutputs` / `signalInputs` 动态生成：

```javascript

        { name: 'power_output',  label: '动力输出', type: 'power', shape: 'circle', color: '#e67e22' },
    ],
    inputs: [
        { name: 'control_channels', label: '控制输入', type: 'control', shape: 'circle', color: '#7f8c8d' },
    ],
}
```

#### ConnectorNode（连接点节点）

```
┌──────────────────────────────┐
│  🔗 left_front_wheel         │
│  (wheel_mount)               │
│──────────────────────────────│
│  ▶ signal_targets            │
│  ▶ power_target              │
│                              │
│  ◀ signal_receive            │
└──────────────────────────────┘
```

### 3.3 状态机节点规格

状态机为本计划的新增图类型，用于编辑零件的动画/行为状态机（如 `idle → running → stall` 等）。

```javascript
// 状态机构造（MachineMax JSON Schema 推演）
"state_machines": {
    "engine_sm": {
        "entry_state": "idle",
        "states": {
            "idle": {
                "animations": ["engine_idle"],
                "transitions": {
                    "start": { "target": "running", "condition": "throttle > 0 && rpm > idle_rpm" },
                    "stall": { "target": "stalled" }
                }
            },
            "running": {
                "animations": ["engine_run"],
                "transitions": {
                    "idle":   { "target": "idle",   "condition": "throttle <= 0 && rpm < idle_rpm + 100" },
                    "stall":  { "target": "stalled", "condition": "rpm < 100" },
                    "redline":{ "target": "over_rev", "condition": "rpm > red_line_rpm" }
                }
            },
            "stalled": {
                "animations": ["engine_stall"],
                "transitions": {
                    "restart": { "target": "idle", "condition": "starter_active" }
                }
            },
            "over_rev": {
                "animations": ["engine_redline"],
                "transitions": {
                    "recover": { "target": "running", "condition": "rpm < red_line_rpm - 200" }
                }
            }
        }
    }
}
```

#### StateNode（状态节点）

```
┌─────────────────────┐
│  💤 idle            │ ← 状态名 + 图标
│  engine_idle        │ ← 关联动画
└─────────────────────┘
```

#### TransitionEdge（转换边）

与信号路由图的普通边不同，转换边附带**条件标签**：

```
  [idle] ─────────( throttle > 0 )─────────→ [running]
                    ^^^ 条件标签显示在边的中点
```

| 端口类型 | 颜色 | 用途 |
|---------|------|------|
| `power` | `#e67e22` 橙 | 机械能/动力传输 |

| `control` | `#2ecc71` 绿 | 控制信号（油门/刹车/转向） |

### 4.2 图数据的持久化

**方案 A（推荐）**：图数据**不独立存储**，每次保存/加载时通过 `toGraph()`/`fromGraph()` 从 MMConfig 重新生成

- ✅ 无数据冗余——唯一的真实来源是 MMConfig
- ✅ 无需版本迁移——图格式随转换器代码演进
- ⚠️ 每次切换标签需重新构建图（但节点数不超过 100，重构建 < 100ms）

**方案 B（备选）**：缓存序列化的图数据到 `MMConfig._uiState._graphCache`

- ❌ 数据冗余，两套格式需保持同步
- ❌ 用户手动改表单后图缓存失效

**选择方案 A**。

### 4.3 双向同步策略

```
用户在节点图中拖拽连线
        ↓
LiteGraph 触发 onConnectionChange 事件
        ↓
调用 fromGraph() 回写到 MMConfig
        ↓
触发 saveConfig() 持久化到 .bbmodel
        ↓
如果属性面板正打开，面板字段同步刷新
```

```
用户在属性面板中修改 signal_targets
        ↓
触发 saveConfig()
        ↓
节点图监听配置变更事件
        ↓
调用 toGraph() 重建图（或增量更新受影响的节点/边）
```

## 5. 状态机编辑器

### 5.1 为什么需要状态机

MachineMax 的动画系统（`animations/*.json`）目前是静态的动画列表。引入状态机后：

- **条件动画切换**：根据游戏状态（`rpm`、`throttle`、`gear` 等）自动切换动画
- **跨子系统联动**：发动机状态改变 → 触发车灯闪烁 → 触发声音切换
- **可组合行为**：共享状态机定义（`subsystem_defs` 中的 `state_machine` 引用）

### 5.2 数据模型

状态机作为一个新的配置域，与当前 `sub_parts[].subsystems[]` 并行：

```javascript
// 在变体级别定义状态机实例
"state_machines": {
    "engine_sm_v8": {
        "definition": "machine_max:engine_state_machine",  // 引用静态定义
        "entry_state": "idle",
        "owner": "subsystem.machine_max.engine",           // 所属子系统
        "states": { ... },
        "transitions": { ... },
    }
}
```

或在 `subsystem_defs` 中定义可复用的状态机模板，架构类似 `connector_defs` / `subsystem_defs`。

> **待确认项**：状态机的确切 JSON Schema 需要与 MachineMax 服务端开发同步确定。本节的数据模型是**基于现有信号系统的推演**，最终实现以服务端 Schema 为准。

### 7.2 节点调色板

侧边栏提供可拖拽的节点模板列表：

- 拖出"发动机"节点到画布 → 创建新的 `SubsystemNode('engine')`
- 拖出"连接点"节点到画布 → 弹出选择对话框（选择已有连接点）
- 拖出"状态"节点到画布 → 创建新的 `StateNode`

### 7.3 交互行为

| 操作 | 行为 |
|------|------|
| 从端口拖拽 | 创建连线 |
| 双击连线 | 编辑连线属性（频道名、条件等） |
| 右键连线 → 删除 | 删除连线 |
| 右键空白 → 添加节点 | 显示节点类型菜单 |
| Ctrl+F | 搜索节点 |
| Ctrl+Z / Ctrl+Y | 撤销 / 重做 |
| 滚轮 | 缩放 |
| 中键拖拽 / 空格+拖拽 | 平移画布 |
| 双击节点 | 打开节点属性面板（侧边 Dock 或 Dialog） |

### 7.4 与属性面板的联动

双击节点时，右侧属性面板切换到该节点对应的配置表单：

```
双击 SubsystemNode("engine")
  → 右侧面板切换到 SubsystemPanel（子系统属性）
  → 设置 subsystemSelection = { spKey, subsystemKey }
  → 面板显示发动机的专属参数

双击 ConnectorNode("left_front_wheel")
  → 选中对应的 Outliner Locator 元素
  → 右侧面板切换到 ConnectorPanel
```

## 11. 风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| LiteGraph.js 与 Blockbench 的 CSS 冲突 | 中 | Canvas 不渲染或样式错乱 | LiteGraph 的可选 CSS 极为精简（主要是 canvas 的 width/height），与 Blockbench 冲突概率低。如有冲突，限定 CSS 作用域到 `.mm-node-graph` 容器 |
| LiteGraph 事件与 Vue 2 响应式冲突 | 低 | 数据不同步 | 所有 LiteGraph 事件回调中使用 `Vue.nextTick()` + `$set` 写入响应式数据 |
| Canvas 渲染在低性能设备上慢 | 低 | 交互卡顿 | 限制同时显示的节点数（分页/虚拟化），自动降级到只读 SVG 模式 |
| 服务端状态机 Schema 变化 | 中 | Phase D 返工 | `from_graph()` 使用适配器模式，Schema 变化只影响适配层 |
| LiteGraph 项目维护停滞 | 低 | 长期维护风险 | LiteGraph 的核心 API 稳定（10 年历史），且源码只有 ~4500 行，fork 维护成本可控 |
| from_graph() 往返一致性 | 中 | 配置丢失或错乱 | 单元测试验证 `to → from → to` 往返 idempotent；Phase C.10 集成测试覆盖所有子系统类型 |
