# 01 — 总体设计

> 最后更新：2026-05-14

***

## 1. 项目概述

### 1.1 项目名称

**MachineMax Blockbench Plugin**（内部代号：`machine_max_bb_plugin`）

### 1.2 一句话描述

在 Blockbench 中新增多个"MachineMax 配置"模式，让 UGC 作者以可视化方式一站式完成 MachineMax 内容包中载具零件、连接点、子系统、材料，以及未来投射物、机甲等全部定义工作。

### 1.3 核心痛点

MachineMax 的内容包制作当前依赖**手工编写 JSON 文件**：

| 痛点    | 说明                                                |
| ----- | ------------------------------------------------- |
| 命名耦合  | 模型中的骨骼名/Locator名必须与 JSON 中的引用精确一致                 |
| 字段繁多  | 仅零件定义 JSON 就有约 40+ 个可配置字段，连接点、子系统各有数十个            |
| 上下文切换 | 作者需要在 Blockbench（建模）和文本编辑器（写 JSON）之间不断切换          |
| 无校验反馈 | 写错字段名或引用不存在的骨骼/Locator，只有进入游戏才能发现                 |
| 学习成本高 | 需要同时掌握 Blockbench + MachineMax JSON Schema + 术语概念 |

### 1.4 目标用户

- **主要用户**：Minecraft 模组 UGC 作者，具备基本建模能力（会用 Blockbench），但缺乏编程/JSON 经验
- **次要用户**：资深内容包作者，需要高效批量配置

***

## 2. 设计原则

| 原则        | 说明                               |
| --------- | -------------------------------- |
| **零命名记忆** | 所有骨骼/Locator 通过可视化选择，不要求用户记忆命名规则 |
| **所见即所得** | 3D 视口中用颜色标注不同用途的元素               |
| **渐进复杂度** | 默认值覆盖 80% 常用场景，高级参数折叠隐藏          |
| **出错即发现** | 引用不存在的骨骼/Locator/资源时即时标红提示       |
| **最小侵入**  | 不修改 Blockbench 核心代码，纯插件实现        |

## 3. 整体架构

### 3.1 三层架构

```
┌─────────────────────────────────────────────────────────┐
│                    表现层 (UI Layer)                     │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────────┐ │
│  │ Outliner │  │  3D 标注     │  │  右侧属性面板     │ │
│  │ (增强)   │  │  (颜色+图标) │  │  (Vue 组件)       │ │
│  └──────────┘  └──────────────┘  └───────────────────┘ │
├─────────────────────────────────────────────────────────┤
│                    业务逻辑层 (Core Layer)               │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────────┐ │
│  │ Part     │  │ Connector    │  │ Subsystem         │ │
│  │ Builder  │  │ Builder      │  │ Builder           │ │
│  └──────────┘  └──────────────┘  └───────────────────┘ │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────────┐ │
│  │ Material │  │ SignalRoute  │  │ Validator         │ │
│  │ Builder  │  │ Builder      │  │ (即时校验)        │ │
│  └──────────┘  └──────────────┘  └───────────────────┘ │
├─────────────────────────────────────────────────────────┤
│                    数据层 (Data Layer)                  │
│  ┌──────────────────┐  ┌──────────────────────────────┐ │
│  │ ConfigStore      │  │ JSON Exporter                │ │
│  │ (内存配置状态)   │  │ (导出为内容包文件)           │ │
│  └──────────────────┘  └──────────────────────────────┘ │
└─────────────────────────────────────────────────────────┘
```

### 3.2 核心模块

#### 3.2.1 表现层

| 模块              | 职责                       | 基于的 BB API                                                  |
| --------------- | ------------------------ | ----------------------------------------------------------- |
| **Outliner 增强** | 右键菜单扩展、自定义图标/颜色、节点筛选     | `OutlinerNode.showContextMenu()`, `Group.icon/Locator.icon` |
| **3D 标注系统**     | 各用途骨骼/Locator 的颜色高亮和图标标注 | `Canvas.gizmos`, `Preview.addAnnotation()`, THREE.js        |
| **属性面板 (Vue)**  | 右侧可折叠表单，根据选中元素类型动态切换内容   | `Mode.component` (Vue 组件)                                   |
| **模式栏**         | 顶部"零件定义"模式按钮，与其他模式并列     | `new Mode(...)`                                             |

#### 3.2.2 业务逻辑层

| 模块                  | 职责 |
|-----------------------|------|
| **ProjectManager**    | 管理多零件项目（零件列表、新增/删除/切换） |
| **PartBuilder**       | 管理零件全局属性（PartType 级字段） |
| **VariantBuilder**    | 管理变体属性（model、textures、tags）和变体间的切换 |
| **SubPartManager**    | 管理子零件的骨骼范围与关联的碰撞箱、连接点、子系统 |
| **ConnectorBuilder**  | 管理连接点实例的配置，引用/创建静态定义 |
| **SubsystemBuilder**  | 管理子系统实例的配置，根据 type 动态显示字段 |
| **MaterialBuilder**   | 管理材料选择与覆写 |
| **SignalRouteBuilder**| 管理信号目标、信号转译和机械能传输的可视化配置（表单 → 拓扑预览 → 交互蓝图，三阶段递进） |
| **Validator**         | 实时校验：骨骼/Locator/资源路径是否存在、必填字段是否缺失 |
| **ElementMarkerManager** | 按 (partId, variantName) 隔离管理元素标记的增删查改 |

#### 3.2.3 数据层

| 模块                 | 职责 |
|----------------------|------|
| **ProjectConfigStore**| 内存中保存 MMProjectConfig（多零件 + 共享定义），支持序列化/反序列化 |
| **JSON Exporter**     | 遍历 parts 和 *_defs，生成完整的 MachineMax 内容包目录结构 |
| **PackFileWriter**    | 管理内容包目录，写入/读取多个 JSON 文件 |

## 4. 技术选型

### 4.1 语言与运行环境

| 项目    | 选择                        | 原因                                           |
| ----- | ------------------------- | -------------------------------------------- |
| 语言    | JavaScript (ES6+)         | Blockbench 插件体系基于 JS，Vue 2 组件用 JS 编写         |
| 类型    | 弱类型（先不用 TypeScript）       | 降低初始开发复杂度；后续可迁移                              |

| UI 框架 | Vue 2（Blockbench 内置的 Vue） | Blockbench 的 Dialog/Mode.component 都基于 Vue 2 |
| 3D 库  | THREE.js（Blockbench 内置）   | Blockbench 的 3D 视口基于 THREE.js                |

### 4.2 关键依赖（Blockbench 内置）

| API / 模块                                                   | 用途              |
| ---------------------------------------------------------- | --------------- |
| `Mode`                                                     | 注册"零件定义"模式      |
| `Action` / `Tool`                                          | 模式专属工具栏按钮       |
| `Dialog`                                                   | 弹出表单对话框         |
| `Menu` / `MenuBar`                                         | 菜单扩展            |

- 所有 UI 基于 Blockbench 内置 Vue 2 + 原生 DOM
- 3D 渲染基于 Blockbench 内置 THREE.js
- 文件操作基于 Electron 内置 Node.js

## 5. 多 Mode 架构

### 5.1 一个插件，多个 Mode

本插件不注册 `ModelFormat`（数据定义 JSON 不包含几何数据），而是在同一个插件内注册多个 `Mode`，各 Mode 共用基础设施：

| Mode ID | 名称 | 适用场景 | 与模型的关系 |
|---------|------|---------|------------|
| `machine_max_part` | 载具零件 | 定义 Part-Variant-SubPart 三级结构 | 依赖 .bbmodel 骨骼/Locator |
| `machine_max_projectile` | 投射物 | 定义投射物弹道参数（未来） | **不依赖** .bbmodel（纯数据编辑） |
| `machine_max_mecha` | 机甲配置 | 机甲/外骨骼配置（未来，待定） | 可能复用 Part 结构 |

**共享基础设施**（跨 Mode 复用）：
- `SubsystemDef` / `ConnectorDef` / `MaterialDef` 编辑器
- 信号路由编辑器（`signal_target_editor`）
- 配置校验器（`validator.js`）
- JSON 导出器框架
- `.bbmodel` Property 持久化

### 5.2 进入"载具零件"Mode

```
用户点击模式栏 [📦 载具零件]
        ↓
Mode.select('machine_max_part') 被调用
        ↓
┌───────────────────────────────────────┐
│ onSelect() 回调执行:                  │
│ 1. Project.machine_max_plugin 已自动 │
│    由 BB Property 恢复到内存          │
│ 2. 取上次编辑的 (partId, variant)     │
│    或默认第一个零件+变体              │
│ 3. 仅加载该变体的 element_markers     │
│ 4. 标注 3D 颜色和图标                 │
│ 5. 渲染右侧属性面板 Vue 组件          │
│ 6. 注册 Outliner 右键菜单扩展         │
│ 7. 显示模式专属工具栏 (含零件导航)    │
└───────────────────────────────────────┘
        ↓
用户在工作（标记元素、配置属性、切换零件/变体）
        ↓
退出模式（切换到 Edit 或关闭文件）
        ↓
onUnselect() 回调执行:
  - 移除 3D 标注
  - 移除 Outliner 右键菜单扩展
  - 销毁 Vue 组件
  - (Property 自动在 BB 保存时持久化，无需手动处理)
```

### 5.3 "投射物"Mode（未来）

不依赖 .bbmodel 骨骼。属性面板为平铺表单（~20 个字段），无需 Outliner 增强和 3D 标注。可在空项目中独立运行。

### 5.4 与其他模式的共存

MachineMax 各 Mode 与 Edit/Paint/Animate 模式**互斥**。进入配置 Mode 时：

- Blockbench 原有的编辑工具不可用（这是"配置模式"，不是"建模模式"）
- 3D 视口变为只读（不可移动/旋转/缩放元素）
- Outliner 变为只读（不可拖拽排序，但可展开/折叠）

## 6. 配置生命周期与持久化

### 6.1 数据流全景

```
┌───────────────────────────────────────────────────────────────────────┐
│                        持久化（自动化）                                │
│                                                                       │
│  ┌──────────┐  Property自动同步  ┌──────────────────┐                │
│  │ .bbmodel │ ←───────────────→ │ Project.machine_  │                │
│  │          │   (打开/保存时)    │ max_plugin        │                │
│  └──────────┘                   └────────┬─────────┘                │
│       │                                  │                            │
│       │ 备选保底                         │ 导出                       │
│       ▼                                  ▼                            │
│  ┌──────────────┐              ┌──────────────────┐                  │
│  │ .bbmodel.    │              │ spark_modules/   │                  │
│  │ mm_project.  │              │ my_pack/         │                  │
│  │ json (备份)  │              │ ├parts/xxx.json  │                  │
│  └──────────────┘              │ ├connectors/...  │                  │
│                                │ ├subsystems/...  │                  │
│                                │ ├projectiles/... │ (未来)            │
│                                │ └materials/...   │                  │
│                                └────────┬─────────┘                  │
│                                         │ 拖入导入                    │
│                                         ▼ (反向解析回填)              │
│                                ┌──────────────────┐                  │
│                                │ 已有内容包 JSON   │                  │
│                                │ parts/hull.json   │                  │
│                                └──────────────────┘                  │
└───────────────────────────────────────────────────────────────────────┘
```

### 6.2 持久化方案：双保险

**主方案 — Blockbench Property 机制**：

在插件加载时注册 `machine_max_plugin` 为 `ModelProject` 的 Property。Blockbench 的 `.bbmodel` compile/parse 循环会**自动**将其纳入保存和加载，无需手动干预。

```javascript
// 在插件 onload 时执行一次
new Property(ModelProject, 'object', 'machine_max_plugin', {
    default: {},
    exposed: false,   // 不在 BB 原生 UI 面板中显示
});
// 之后 Project.machine_max_plugin 的读写都会自动随 .bbmodel 持久化
```

**备选 — 独立配置文件**：

每次保存时额外写入 `.bbmodel.mm_project.json`，用于以下场景：
- 插件未加载时用户误操作 Ctrl+S 导致 Property 未写入（极罕见）
- 跨机迁移时作为保底恢复数据源

**加载优先级**：
```
打开 .bbmodel → Property 自动恢复到 Project.machine_max_plugin
  ├── 存在且有效 → ✅ 主方案命中
  └── 不存在/无效 → 尝试读取 .bbmodel.mm_project.json
       ├── 存在 → 回填到 Project.machine_max_plugin ✅
       └── 不存在 → 创建空白 MMProjectConfig ✅
```

### 6.3 导出

点击"导出"按钮，遍历 `parts`、`projectiles`（未来）和 `*_defs` 写为完整内容包目录结构。

### 6.4 拖入导入（反向解析）

用户可将已有的内容包 JSON 文件（如 `parts/hull.json`、`connectors/wheel_mount.json`）拖入 Blockbench 窗口，插件自动反向解析并回填到 `MMProjectConfig`：
- 拖入 `parts/*.json` → 解析 PartType 结构 → 写入 `MMProjectConfig.parts`
- 拖入 `connectors/*.json` → 解析 ConnectorAttr → 写入 `connector_defs`
- 拖入 `subsystems/*.json` → 解析 SubsystemAttr → 写入 `subsystem_defs`

这使得手工编写过 JSON 的用户可以无缝迁移到可视化编辑。

### 6.5 关键设计要点

- **多零件管理**：一个模型 → 多个零件定义（如 wine_fox 有 11 个 part），每个零件有独立变体
- **跨零件共享**：`connector_defs`、`subsystem_defs`、`material_defs` 被所有零件和类型共享
- **校验**：导出前自动校验所有必填字段、引用完整性和 Part-Variant-SubPart 结构

## 7. 内容范围

### 7.1 本插件覆盖的内容包模块

| 模块                                           | 覆盖程度     | 说明                        |
| -------------------------------------------- | -------- | ------------------------- |
| `parts/*.json`                               | **完全覆盖** | 零件定义的全部字段                 |
| `connectors/*.json`                          | **完全覆盖** | 连接点静态定义的全部字段              |
| `subsystems/*.json`                          | **完全覆盖** | 子系统静态定义的全部字段              |
| `materials/*.json`                           | **部分覆盖** | 材料定义（第一版可限于选择已有材料，后续支持新建） |
| `lang/*.json`                                | **基本覆盖** | 自动生成本地化条目标识符              |
| `meta.json`                                  | **基本覆盖** | 可在插件中创建/编辑                |
| `models/`                                    | **不覆盖**  | 建模在 Blockbench 原有模式下完成    |
| `textures/`                                  | **不覆盖**  | 贴图在 Blockbench 原有模式下完成    |
| `animations/`                                | **不覆盖**  | 动画在 Animate 模式下完成         |
| `projectiles/*.json`                          | **后续版本** | 投射物定义（待 MachineMax 武器系统正式发布后跟进） |
| `recipe/`                                    | 后续版本     | 配方定义                      |
| `templates/` / `blueprints/` / `assemblies/` | 后续版本     | 结构数据                      |

| `machine_max:item_storage` | 物品存储 | 可存储物品，可指定容量 |
| `machine_max:motor` | 电动机 | 电力驱动，与发动机类似的动力输出 |
| `machine_max:seat` | 座位 | 乘坐位置、信号输出 |
| `machine_max:transmission` | 传动系统 | 将转速/转矩分发给各轴 |

## 8. 非功能需求

| 需求  | 指标                                             |
| --- | ---------------------------------------------- |
| 性能  | 百级零件配置下，模式切换 < 1 秒                             |
| 兼容性 | Blockbench ≥ 4.11（与当前 `blockbench` 目录版本一致）     |
| 平台  | 仅桌面版（Windows/macOS/Linux），需 Electron + Node.js |
| 语言  | 界面为中文（目标用户群体），代码注释为中文                          |
| 容错  | 错误的骨骼/Locator 引用在 3D 视口中标红                     |
| 撤销  | 配置修改应支持 Blockbench 原生 Undo 系统（尽可能）             |

## 9. 风险与缓解

| 风险                        | 概率 | 缓解措施                                       |
| ------------------------- | -- | ------------------------------------------ |
| `.bbmodel` 保存时丢弃 extra 字段 | 低  | Blockbench Property 机制已通过源码验证：注册为 `ModelProject` Property 后自动纳入 compile/parse 循环 |
| Property 未注册时丢失数据 | 低  | 双保险：每次保存额外写入 `.bbmodel.mm_project.json` 独立备份 |
| Vue 2 API 与插件模式不兼容        | 低  | BB 的 Dialog/Mode.component 均基于 Vue 2，验证过可用 |
| 桌面版 fs 模块受限               | 低  | Electron 不限制 Node.js API；需处理权限问题           |
| 字段变更导致不同步                 | 中  | 导出时用最新 Schema 校验；版本号关联                     |
| 用户弃用（上手成本仍高）              | 中  | 分阶段交付，每阶段可独立使用；默认值覆盖常用场景                   |

## 10. 未来类型扩展

### 10.1 机甲/外骨骼

如果机甲本质上是一种特殊载具（持有 Part-Variant-SubPart 结构），直接复用 `machine_max_part` Mode，用 `tags: ["machine_max:mecha"]` 区分。只有数据模型与当前 Part 三级结构有本质不同时，才需要新的 Mode。

### 10.2 投射物

投射物 Mode 的特点：
- 纯数据编辑，**不关联 3D 骨骼**（无 Outliner 增强、无 3D 标注）
- 平铺表单（~20 个字段：弹道参数、速度-伤害模型系数）
- **可在空项目中使用**（不需要打开 .bbmodel）
- 但共享子系统编辑器基础设施（若投射物也持有子系统）

### 10.3 扩展策略总结

```
MachineMax Blockbench Plugin (一个插件)
│
├── Mode: 载具零件 (machine_max_part)    ← 阶段一~六实现
├── Mode: 投射物 (machine_max_projectile) ← 待武器系统发布后跟进
└── Mode: 机甲 (machine_max_mecha)        ← 待定，可能复用 Part Mode

所有 Mode 共享:
  ├── SubsystemDef / ConnectorDef / MaterialDef 编辑器
  ├── 信号路由编辑器（表单 + 拓扑预览 + 蓝图）
  ├── 校验器 + JSON 导出器
  └── Project.machine_max_plugin Property 持久化
```

## 11. 文档约定

- **BB** = Blockbench
- **MM** = MachineMax
- **UGC** = User-Generated Content（用户创作内容）
- **Outliner** = Blockbench 的大纲视图
- **Locator** = Blockbench 模型中的定位器（空节点）
- **Group** = Blockbench 模型中的组/骨骼节点
