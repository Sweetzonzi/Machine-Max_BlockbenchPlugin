## 2. Outliner 右键菜单扩展（方案C）

> **2026-05-15 更新**：经过多轮调试，此方案已完整验证并修复。以下为关键发现和最终实现。

### 2.1 继承层次

Blockbench 5.0.7 中 Outliner 元素的继承关系：

```
OutlinerNode                 ← 基类，有 showContextMenu
├── Group                    ← 重写 showContextMenu，不调 super
└── OutlinerElement          ← 重写 showContextMenu，不调 super
    └── Locator (继承此方法)
    └── Cube (继承此方法)
```

**关键点**：Group 和 OutlinerElement 各自**重写**了 `showContextMenu`，且都**不调用 super()**。因此仅劫持 `OutlinerNode.prototype` 对 Group/Locator/Cube 完全无效。

### 2.2 最终实现：三原型劫持 + structure 注入/弹出

```javascript
// 需要劫持的三个原型（按名称延迟求值，避免 IIFE 加载时全局变量未就绪）
const PATCH_TARGETS = ['OutlinerNode', 'Group', 'OutlinerElement'];

// 劫持模式：在 open() 前推入、open() 后弹出
capturedProto.showContextMenu = function(event) {
    // 1. 注入 MM 条目到 this.menu.structure（在 open() 渲染前完成）
    var injectedCount = injectMMItemsToStructure(this, this.menu);

    // 2. 调用原始 showContextMenu → 内部 open() 读取 structure 渲染菜单
    originalFn.call(this, event);

    // 3. open() 返回后弹出条止，恢复 structure 原样
    ejectMMItemsFromStructure(this.menu, injectedCount);
};
```

### 2.3 为什么不能新建独立 Menu

Blockbench 一次只允许打开一个 `open_menu`。若在标准菜单打开后创建独立 `Menu.show(event)`，会先从 DOM 移除第一个菜单，导致两个菜单互斥。

### 2.4 Menu 没有 addItem 方法

`Group.prototype.menu` 和 `Locator.prototype.menu` 是**所有同类元素共享**的预构建 `Menu` 实例：

```javascript
// group.js:492
Group.prototype.menu = new Menu([...静态条目数组...]);
```

这个 Menu 构造函数接收 `structure` 数组，**没有** `addItem()` 方法。要追加条目只能往 `menu.structure` 推入。

### 2.5 没有 MenuItem 类

Blockbench **不存在** `MenuItem` 类！菜单条目直接使用**普通对象描述符**：

```javascript
// ❌ 错误 — MenuItem 不存在
new MenuItem('标题', { onClick: ... })

// ✅ 正确 — 普通对象
{ name: '标题', icon: 'icon_name', click: function() { ... } }
```

全局可用的有 `Menu`、`MenuSeparator`、`BarMenu`，但**没有** `MenuItem`。

参考：[menu.js](file:///d:/Files/Project_MinecraftMods/blockbench/js/interface/menu.js)，条目解析见 `getEntry()` → `populateList()`。

## 6. Outliner 面板在自定义模式下显示

### 6.1 Outliner 是 Panel，不是 BarItem

```javascript
Panels.outliner.update();

// 退出模式时
var idx = cond.modes.indexOf('machine_max_part');
```

## 9. .bbmodel 持久化（Property 机制 + 双保险）

### 9.1 原理

Blockbench 的 `.bbmodel` compile/parse 遍历 `ModelProject.properties` 注册表（[bbmodel.js:L201-L204](file:///d:/Files/Project_MinecraftMods/blockbench/js/formats/bbmodel.js#L201-L204)）。注册为 Property 后自动纳入保存/加载，无需手动干预。

对 object 类型 Property，`copy()` 使用 `structuredClone`（深拷贝，property.ts:L217-219），`merge()` 使用 `structuredClone`（深拷贝恢复，property.ts:L189-191）。

### 9.2 实现代码

```javascript
// ===== utils/persistence.js =====

const CONFIG_VERSION = 3;
const PROPERTY_NAME = 'machine_max_plugin';

/**
 * 在插件 onload 时调用。注册 Property 到 ModelProject，
 * Blockbench 自动将 Project[PROPERTY_NAME] 纳入 .bbmodel 的保存/加载循环。
 */
export function registerProperty() {
    if (ModelProject.properties[PROPERTY_NAME]) {
        return; // 已注册（热重载场景）
    }
    new Property(ModelProject, 'object', PROPERTY_NAME, {
        default: {},
        exposed: false,
    });
}

/**
 * 从已恢复的 Project.machine_max_plugin 或备选文件加载配置。
 * 应在 Mode.onSelect 时调用。
 */
export function loadConfig(bbmodelPath) {
    // 1. 主方案：Property 已由 BB 自动恢复到内存
    if (Project[PROPERTY_NAME]?.$schema_version) {
        return migrateIfNeeded(Project[PROPERTY_NAME]);
    }

    // 2. 备选：独立文件保底
    const standalonePath = bbmodelPath.replace(/\.bbmodel$/, '.mm_project.json');
    if (fs.existsSync(standalonePath)) {
        try {
            const raw = JSON.parse(fs.readFileSync(standalonePath, 'utf-8'));
            const config = raw.config || raw;
            Project[PROPERTY_NAME] = migrateIfNeeded(config);
            return Project[PROPERTY_NAME];
        } catch (e) {
            console.warn('[MM Plugin] 备选配置文件读取失败:', e);
        }
    }

    // 3. 全新项目
    const blank = createBlankConfig();
    Project[PROPERTY_NAME] = blank;
    return blank;
}

/**
 * 保存配置。Property 自动纳入 Ctrl+S，这里额外写独立备份。
 * 应在 Mode.onUnselect 时调用，或监听 BB 保存事件。
 */
export function saveConfig(bbmodelPath, config) {
    // 主方案：直接赋值，Property 自动纳入下次 Ctrl+S
    Project[PROPERTY_NAME] = config;

    // 备选：独立文件保底
    const standalonePath = bbmodelPath.replace(/\.bbmodel$/, '.mm_project.json');
    try {
        fs.writeFileSync(
            standalonePath,
            JSON.stringify({
                $schema_version: CONFIG_VERSION,
                bbmodel: path.basename(bbmodelPath),
                timestamp: Date.now(),
                config: config,
            }, null, 2),
            'utf-8'
        );
    } catch (e) {
        console.warn('[MM Plugin] 备选配置写入失败:', e);
    }
}

function createBlankConfig() {
    return {
        $schema_version: CONFIG_VERSION,
        namespace: 'machine_max',
        parts: {},
        projectiles: {},
        connector_defs: {},
        subsystem_defs: {},
        material_defs: {},
    };
}
```

## 12. 拖入 JSON 导入

用户可将已有内容包 JSON 文件拖入 Blockbench 窗口进行反向解析回填。

```javascript
// ===== importers/part_importer.js =====

/**
 * 从 parts/*.json 反向解析并回填到 MMProjectConfig.parts
 */
export function importPartJSON(jsonPath, jsonContent) {
    const part = jsonContent;

    // 从文件名推断 partId
    const partId = path.basename(jsonPath, '.json');

    // 解析顶层字段
    const config = {
        icon: part.icon || '',
        vehicle_durability_rate: part.vehicle_durability_rate ?? 0.8,
        vehicle_damage_rate: part.vehicle_damage_rate ?? 1.0,
        vehicle_damage_rate_destroyed: part.vehicle_damage_rate_destroyed ?? 0.1,
        functional_threshold: part.functional_threshold ?? 0.3,
        share_durability: part.share_durability ?? true,
        max_stack_size: part.max_stack_size ?? 1,
        variants: {},
        element_markers: {},
    };

    // 解析变体（处理简写/完整格式）
    if (part.variants) {
        const variantsRaw = part.variants;
        if (variantsRaw.model) {
            // 简写格式：只有一个 default 变体
            config.variants['default'] = parseVariant(variantsRaw);
        } else {
            for (const [name, variant] of Object.entries(variantsRaw)) {
                config.variants[name] = parseVariant(variant);
            }
        }
    }

    MMProjectConfig.parts[partId] = config;
    return partId;
}

function parseVariant(variant) {
    const subParts = {};
    const subPartsRaw = variant.sub_parts;
    if (subPartsRaw) {
        if (subPartsRaw.start_bone !== undefined) {
            subParts['sub_part.machine_max.main'] = parseSubPart(subPartsRaw);
        } else {
            for (const [name, sp] of Object.entries(subPartsRaw)) {
                subParts[name] = parseSubPart(sp);
            }
        }
    }
    const textures = variant.textures;
    let parsedTextures = {};
    if (typeof textures === 'string' || Array.isArray(textures)) {
        parsedTextures = textures;
    } else if (typeof textures === 'object') {
        parsedTextures = textures;
    }
    return {
        model: variant.model || '',
        textures: parsedTextures,
        animations: variant.animations || null,
        tags: variant.tags || [],
        sub_parts: subParts,
    };
}

function parseSubPart(sp) {
    return {
        start_bone: sp.start_bone || '',
        end_bones: sp.end_bones || [],
        durability: sp.durability ?? 20.0,
        mass: sp.mass ?? 25.0,
        mass_center: sp.mass_center || 'mass_center',
        projected_area: sp.projected_area || [0, 0, 0],
        block_collision: sp.block_collision || 'true',
        collision_height: sp.collision_height ?? -1.0,
        climb_assist: sp.climb_assist ?? false,
        hydro_priority: sp.hydro_priority ?? 0,
        hit_boxes: sp.hit_boxes || {},
        interact_boxes: sp.interact_boxes || {},
        connectors: sp.connectors || {},
        subsystems: sp.subsystems || {},
        hydrodynamics: sp.hydrodynamics || null,
    };
}
```

| 尝试使用的 API | 是否存在 | 正确替代 |
|:---|:---:|:---|
| `Action.actions` | ❌ | `BarItems` (Record\<string, BarItem\>) |
| `Action.actions[id].delete()` | ❌ | `BarItems[id].delete()` |
| `BarItems?.Mode?.getSelectedId?.()` | ❌ | `Modes.id` |
| `BarItems.outliner` | ❌ | `Panels.outliner` (Panel 实例) |
| `BarItems.outliner.open()` | ❌ | 修改 `Panels.outliner.condition.modes` |
| `new MenuItem(...)` | ❌ | 普通对象 `{name, icon, click}` |
| `menu.addItem(...)` | ❌ | `menu.structure.push(...)` |
| `new Dialog({...})` — 自动显示 | ❌ | `new Dialog({...}).show()` |
| `Group.prototype.showContextMenu` 能覆盖所有元素 | ❌ | 需同时劫持 OutlinerNode/Group/OutlinerElement 三个原型 |
| `Outliner.selected` 包含 Group | ❌ | Group 继承自 `OutlinerNode`（非 `OutlinerElement`），其 `markAsSelected` 为空方法。选中 Group 时仅子 Cube 进入 `Outliner.selected`。应使用 `Group.first_selected` |
| `Mode.component` 随模式切换自动隐藏 | ❌ | `Mode.component` 挂载到 `#center`，切换模式不销毁。应改用 `Panel` + `condition: {modes: [...]}` 实现自动显隐 |
| `Project.loaded` | ❌? | 可能不存在。改用 `Project === 0` 判断无项目状态 |
| `new Property(...)` 后 Project[prop] 保持原值 | ❌ | `new Property(obj, 'object', name, {default: {}})` 会将 `obj[name]` **重置为 default**，覆盖 BB 已恢复的保存数据。必须在注册前备份并在注册后还原 |
| `migrateIfNeeded` 返回的对象与 `Project[prop]` 是同一引用 | ❌ | `ensureDefaults` 内部用 `Object.assign({}, blank, config)` 创建新对象，**不是同一引用**。`loadConfig()` 的 Property 路径必须显式写回 `Project[PROPERTY_NAME] = migrated`，否则 `onSelect` 中对 `_uiState` 的修改无法被 Vue 面板的 `getConfig()` 读取到 |

## 14. 信号路由数据模型（提取信号拓扑图）

### 14.1 数据来源

信号拓扑图的数据来源是 `MMConfig` 中**当前零件当前变体**的配置。需要提取以下三类路由配置：

#### 14.1.1 连接点信号路由

```javascript
// 遍历当前变体所有子零件的 connectors
function extractConnectorSignals(variant) {
    const edges = [];
    const nodes = [];

    for (const [spName, sp] of Object.entries(variant.sub_parts || {})) {
        for (const [connName, conn] of Object.entries(sp.connectors || {})) {
            // 每个连接点是一个节点
            nodes.push({
                id: connName,
                type: 'connector',
                subPart: spName,
                locator: conn.locator,
            });
            // signal_targets: { 频道名 → [目标列表] }
            for (const [channel, targets] of Object.entries(conn.signal_targets || {})) {
                for (const target of targets) {
                    edges.push({
                        from: connName,
                        to: target,
                        channel: channel,
                        type: 'signal',
                    });
                }
            }
            // power_target: 机械能输出目标（单值）
            if (conn.power_target) {
                edges.push({
                    from: connName,
                    to: conn.power_target,
                    channel: 'power',
                    type: 'power',
                });
            }
        }
    }
    return { nodes, edges };
}
```

#### 14.1.2 子系统信号路由

```javascript
function extractSubsystemSignals(variant) {
    const edges = [];
    const nodes = [];

    for (const [spName, sp] of Object.entries(variant.sub_parts || {})) {
        for (const [subName, sub] of Object.entries(sp.subsystems || {})) {
            nodes.push({
                id: subName,
                type: 'subsystem',
                subPart: spName,
                subType: sub.type,
            });
            // control_outputs: { 频道名 → [目标列表] }
            for (const [channel, targets] of Object.entries(sub.control_outputs || {})) {
                for (const target of targets) {
                    edges.push({ from: subName, to: target, channel, type: 'control' });
                }
            }
            // speed_outputs: { 频道名 → [目标列表] }
            for (const [channel, targets] of Object.entries(sub.speed_outputs || {})) {
                for (const target of targets) {
                    edges.push({ from: subName, to: target, channel, type: 'speed' });
                }
            }
            // power_output: 机械能输出目标（单值）
            if (sub.power_output) {
                edges.push({ from: subName, to: sub.power_output, channel: 'power', type: 'power' });

        }
    }
    return { nodes, edges };
}
```

### 14.2 特殊目标节点的处理

`signal_targets` 的目标可以指向四种类型之一的路径：

- `"subsystem.命名空间.名称"` — 指向子系统
- `"connector.命名空间.名称"` — 指向连接点
- `"subpart"` — 特殊常量，指向所属子零件自身
- `"vehicle"` — 特殊常量，指向载具级信号存储

在拓扑图中，`"subpart"` 和 `"vehicle"` 作为特殊节点渲染（虚线框，灰色），其他目标直接通过 `resolveSignalTarget()` 查找并引向对应的真实节点。

### 14.3 信号转译的表示

`signal_translations` 在图中不作为独立节点，而是作为**连线上的标签转换标注**。当连线经过配置了 `signal_translations` 的连接点时：

```
输入频道: steer ← 连接点 { translations: { steer: "steering" } } → 输出频道: steering
```

在绘图中表现为连线上多出一个菱形标注框，内写源频道 → 目标频道。

## 15. SVG 信号拓扑图渲染方案

### 15.1 技术选型理由

| 方案 | 优点 | 缺点 |
|------|------|------|
| **SVG + DOM**（选用） | Vue 响应式天然兼容；点击/悬停事件与 DOM 一致；无 Canvas hit-testing | 节点数 > 100 时性能下降 |
| **Canvas 2D** | 渲染性能更高 | Hit-testing 需自行实现；无响应式绑定 |
| **Canvas + WebGL** | 适合大量节点 | 过度设计，信号拓扑通常 < 50 节点 |
| **纯 DOM + CSS**（动画控制器方案） | 开发最快 | 无法自由布局，仅支持线性排列 |

选择 SVG 的理由：信号拓扑的节点数通常在 10～50 之间，SVG 性能完全足够，且与 Vue 2 响应式配合最自然。

### 15.2 简易力导向布局算法

不引入 d3-force 依赖（避免 esbuild 打包问题），自实现简化版：

```javascript
function forceLayout(nodes, edges, options) {
    const { width = 600, height = 400,
            repulsion = 1000, attraction = 0.01,
            iterations = 100, damping = 0.9 } = options;

    // 初始化：随机位置
    for (const node of nodes) {
        if (!node.x) node.x = Math.random() * width;
        if (!node.y) node.y = Math.random() * height;
        node.vx = 0; node.vy = 0;
    }

    for (let iter = 0; iter < iterations; iter++) {
        // 电荷斥力：所有节点对之间（库仑定律）
        for (let i = 0; i < nodes.length; i++) {
            for (let j = i + 1; j < nodes.length; j++) {
                const dx = nodes[j].x - nodes[i].x;
                const dy = nodes[j].y - nodes[i].y;
                const dist = Math.max(Math.sqrt(dx*dx + dy*dy), 1);
                const force = repulsion / (dist * dist);
                nodes[i].vx -= force * (dx / dist);
                nodes[i].vy -= force * (dy / dist);
                nodes[j].vx += force * (dx / dist);
                nodes[j].vy += force * (dy / dist);
            }
        }

        // 弹簧引力：有边连接的节点之间（胡克定律）
        for (const edge of edges) {
            const source = nodes.find(n => n.id === edge.from);
            const target = nodes.find(n => n.id === edge.to);
            if (!source || !target) continue;
            const dx = target.x - source.x;
            const dy = target.y - source.y;
            const dist = Math.sqrt(dx*dx + dy*dy);
            const force = attraction * (dist - 100); // 期望距离 100px
            source.vx += force * (dx / dist);
            source.vy += force * (dy / dist);
            target.vx -= force * (dx / dist);
            target.vy -= force * (dy / dist);
        }

        // 中心引力（防止散开）
        const cx = width / 2, cy = height / 2;
        for (const node of nodes) {
            node.vx += (cx - node.x) * 0.001;
            node.vy += (cy - node.y) * 0.001;
        }

        // 更新位置（加入阻尼）
        for (const node of nodes) {
            node.vx *= damping;
            node.vy *= damping;
            node.x += node.vx;
            node.y += node.vy;
            // 边界限制
            node.x = Math.max(20, Math.min(width - 20, node.x));
            node.y = Math.max(20, Math.min(height - 20, node.y));
        }
    }
    return nodes;
}
```

### 15.4 连线路径计算（正交折线）

```javascript
function computeOrthogonalPath(from, to, nodes) {
    const x1 = from.x + nodeWidth;
    const y1 = from.y + nodeHeight / 2;
    const x2 = to.x;
    const y2 = to.y + nodeHeight / 2;

    // 简单折线：水平→垂直→水平
    const midX = (x1 + x2) / 2;

    // 检查折线是否与其他节点相交
    const waypoints = [
        { x: x1, y: y1 },
        { x: midX, y: y1 },
        { x: midX, y: y2 },
        { x: x2, y: y2 },
    ];

    // 简化：如果有节点遮挡，增加绕行段
    for (const node of nodes) {
        if (intersects(node, waypoints)) {
            // 向上或向下偏移中间段
            waypoints[1].y = y1 - offset;
            waypoints[2].y = y2 - offset;
        }
    }

    return waypoints.map((p, i) =>
        (i === 0 ? 'M' : 'L') + p.x.toFixed(1) + ' ' + p.y.toFixed(1)
    ).join(' ');
}
```

## 16. Blockbench 动画控制器与信号拓扑图的差异

### 16.1 核心差异对比

| 维度 | 动画控制器（BB 内置） | 信号拓扑图（本插件） |
|------|---------------------|-------------------|
| **数据结构** | 线性链表（状态 → 下一个状态） | **有向图**（多对多，含回路） |
| **节点排列** | 水平一维（CSS Flexbox） | 二维自由布局 |
| **位置计算** | 由数组索引决定：`i * 312px` | 力导向布局算法 |
| **连线绘制** | CSS border 拱形 | SVG `<path>` 正交折线 |
| **端口** | 无端口概念 | 输入/输出端口（左右侧） |
| **画布缩放** | CSS `zoom` 属性（0.3x～1.0x） | SVG `transform: scale()` |
| **画布平移** | 不支持 | 拖拽平移视口 |
| **节点拖拽** | 不支持（仅排序） | 支持自由拖拽 |
| **连线交互** | 仅双击编辑文字 | 拖拽创建 + 右键删除 + 悬停高亮 |
| **多目标** | 不支持 | 一个频道 → 多个目标 |

### 16.2 结论

动画控制器的 DOM + CSS 实现方式在**低维度线性状态机**场景下高效且轻量，但完全不适合**有向图**形式的信号路由可视化。信号拓扑图必须使用 SVG 或 Canvas 从零构建，无法基于动画控制器扩展。

## 17. Phase 1 连接点面板的信号路由表单实现要点

### 17.1 关键约束

1. **`signal_targets` 是 `频道名 → [目标数组]`**：一个频道名（如 `move_input`）可以指向多个目标。UI 上需在一个频道条目内提供可增减的目标列表。
2. **目标自动补全的范围**：当前零件内所有子系统的 key 名、所有连接点的 key 名、特殊常量 `"subpart"` 和 `"vehicle"`。
3. **频道名预设**：`move_input`、`regular_input`、`engine_control`、`wheel_control`、`engine_speed`、`steering`、`interact` 等。预设值应在下拉框顶部，并允许用户输入自定义值。
4. **`signal_translations` 是 `源频道 → 目标频道` 的 1:1 映射**：无需目标列表，直接使用两列键值对。
5. **`overwrite` 是可折叠区域**：只显示在 `definition` 中已存在的字段，并标记当前值和默认值。

### 17.2 数据绑定

连接点实例的 `connectors[connName]` 对象中：

### 18.3 默认名称生成规则

```javascript
/**
 * 根据实体类型和上下文生成默认名称（翻译键格式）
 * @param {'sub_part'|'connector'|'subsystem'} entityType
 * @param {Object} context - 生成上下文
 * @param {string} context.namespace - 命名空间，默认 'machine_max'
 * @param {string} context.boneName - 骨骼/Locator 名称（用于 sub_part/connector）
 * @param {string} context.typeShortName - type 短名（用于 subsystem）
 * @param {number} context.index - 序号（用于 subsystem 去重）
 * @returns {string} 默认名称
 */
function generateDefaultName(entityType, context) {
  const ns = context.namespace || 'machine_max';

  switch (entityType) {
    case 'sub_part':
      // 子零件默认：sub_part.<ns>.main，如已存在则加后缀
      return `sub_part.${ns}.main`;

    case 'connector':
      // 连接点默认：connector.<ns>.<locator名转snake_case>
      const locName = context.boneName || 'connector';
      return `connector.${ns}.${toSnakeCase(locName)}`;

    case 'subsystem':
      // 子系统默认：subsystem.<ns>.<type短名>_<序号>
      const shortName = context.typeShortName || 'subsystem';
      const idx = context.index || 1;
      return `subsystem.${ns}.${shortName}_${idx}`;

    default:
      return '';
  }
}

function toSnakeCase(str) {
  // "LeftFrontWheel" → "left_front_wheel"
  return str
    .replace(/([A-Z])/g, '_$1')
    .toLowerCase()
    .replace(/^_/, '');
}
```

### 18.4 唯一性校验

```javascript
/**
 * 校验名称是否唯一
 * @param {Object} variant - 当前变体配置
 * @param {'sub_part'|'connector'|'subsystem'} scope - 校验作用域
 * @param {string} subPartKey - 当前子零件 key（connector/subsystem 需此参数）
 * @param {string} name - 要校验的名称
 * @returns {{ valid: boolean, message: string }}
 */
function validateNameUniqueness(variant, scope, subPartKey, name) {
  if (!variant || !name) return { valid: false, message: '名称为空' };

  let existingKeys = [];

  switch (scope) {
    case 'sub_part':
      // 同一变体下所有子零件
      existingKeys = Object.keys(variant.sub_parts || {});
      break;
    case 'connector':
      // 同一子零件下所有连接点
      const spConns = variant.sub_parts?.[subPartKey]?.connectors || {};
      existingKeys = Object.keys(spConns);
      break;
    case 'subsystem':
      // 同一子零件下所有子系统
      const spSubs = variant.sub_parts?.[subPartKey]?.subsystems || {};
      existingKeys = Object.keys(spSubs);
      break;
  }

  if (existingKeys.includes(name)) {
    return { valid: false, message: `"${name}" 已存在，请换一个名称` };
  }

  return { valid: true, message: '' };
}
```

### 18.5 重命名同步流程

当用户在 UI 中修改实体名称时，需执行完整的同步操作链：

```javascript
/**
 * 重命名子零件并同步关联数据
 * @param {Object} variant - 当前变体
 * @param {string} oldKey - 旧名称
 * @param {string} newKey - 新名称
 * @param {Object} partData - 零件配置（含 element_markers）
 * @param {string} variantName - 变体名
 */
function renameSubPart(variant, oldKey, newKey, partData, variantName) {
  const spConfig = variant.sub_parts[oldKey];
  if (!spConfig) return;

  // 1. 迁移字典
  this.$set(variant.sub_parts, newKey, spConfig);
  this.$delete(variant.sub_parts, oldKey);

  // 2. 更新 element_markers 中的 config_ref
  const markers = partData.element_markers?.[variantName] || {};
  for (const [uuid, marker] of Object.entries(markers)) {
    if (marker.type === 'sub_part' && marker.config_ref === oldKey) {
      this.$set(markers, uuid, { ...marker, config_ref: newKey });
    }
  }

  // 3. 更新子零件自身的 start_bone（仅当 start_bone 与旧 key 相同时）
  //    注意：如果用户已手动改了 start_bone 为不同的值，则不自动更新
  if (spConfig.start_bone === oldKey) {
    spConfig.start_bone = newKey;
  }

  // 4. 重算 auto_end_bones
  recalcAutoEndBones(partData, variantName, variant.sub_parts);

  // 5. 刷新 Outliner 图标
  refreshOutlinerIcons();
}

/**
 * 重命名连接点并同步关联数据
 * @param {Object} subPartConfig - 子零件配置
 * @param {string} oldKey - 旧名称
 * @param {string} newKey - 新名称
 */
function renameConnector(subPartConfig, oldKey, newKey) {
  const connConfig = subPartConfig.connectors[oldKey];
  if (!connConfig) return;

  this.$set(subPartConfig.connectors, newKey, connConfig);
  this.$delete(subPartConfig.connectors, oldKey);
  // 注：element_markers 的更新由调用方负责
}
```
