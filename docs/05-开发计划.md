## 1. 开发策略

采用**分阶段交付**策略，每个阶段产出可独立使用的功能。阶段划分已根据当前代码现状重新调整：

```
阶段一 ──→ 阶段二 ──→ 阶段三 ──→ 阶段四 ──→ 阶段五 ──→ 阶段六
(骨架)     (连接点)   (子系统)   (信号路由   (信号拓扑   (导出与增强)
                                表单)       预览+蓝图)
```

每阶段完成后即可交付给 UGC 作者试用并提供反馈。

## 6. 阶段四：信号路由表单（Phase 1）

### 6.1 背景与动机

信号系统是 MachineMax 连接所有子系统的通信网络。在 JSON 中手动指定信号流向对无代码基础的 UGC 作者难度极大且容易出错。

根据 [5.4-连接点定义.md](file:///d:/Files/Project_MinecraftMods/Machine-Max/docs/wiki/5-%E5%86%85%E5%AE%B9%E5%8C%85%E5%88%B6%E4%BD%9C%E6%95%99%E7%A8%8B/5.4-%E8%BF%9E%E6%8E%A5%E7%82%B9%E5%AE%9A%E4%B9%89.md) 和 [3.6-信号系统.md](file:///d:/Files/Project_MinecraftMods/Machine-Max/docs/wiki/3-%E5%AD%90%E7%B3%BB%E7%BB%9F%E8%AF%A6%E8%A7%A3/3.6-%E4%BF%A1%E5%8F%B7%E7%B3%BB%E7%BB%9F.md)，信号路由配置分布在以下三处：

| 配置点 | JSON 路径 | 复杂度 |
|--------|----------|--------|
| 连接点实例 | `sub_parts[].connectors[].signal_targets` | 中 |
| 连接点实例 | `sub_parts[].connectors[].signal_translations` | 中 |
| 连接点实例 | `sub_parts[].connectors[].power_target` | 低 |
| 连接点实例 | `sub_parts[].connectors[].internal` | 低 |
| 连接点实例 | `sub_parts[].connectors[].overwrite` | 中 |
| 连接点实例 | `sub_parts[].connectors[].definition` | 低（已有 UI） |
| 子系统实例 | `sub_parts[].subsystems[].control_outputs` | 高 |
| 子系统实例 | `sub_parts[].subsystems[].speed_outputs` | 中 |
| 子系统实例 | `sub_parts[].subsystems[].power_output` | 低 |
| 交互盒 | `sub_parts[].interact_boxes[].signal_targets` | 中 |

**当前插件覆盖情况**：连接点的 `definition` 和 `locator` 已有 UI，其余全部为 0。

### 6.2 目标

补齐**所有信号路由相关的表单 UI**，让 UGC 作者无需手写 JSON 即可完成信号配置。本阶段不做可视化图编辑器，而是用**完善的表单组件**覆盖所有配置字段，配合自动补全和即时校验确保不出错。

### 6.3 设计原则

- **键值对列表编辑器**：`signal_targets`、`signal_translations`、`control_outputs` 等映射关系统一使用可增删的键值对列表 UI
- **自动补全**：所有目标引用（子系统名、连接点名、`"subpart"`、`"vehicle"`）提供下拉补全
- **频道名预设**：常用频道名（`move_control`、`engine_control`、`wheel_control` 等）提供预设选择和自定义输入
- **即时校验**：引用不存在的目标时标红提示
- **覆写展开**：`overwrite` 字段作为可折叠区域，默认收起

## 7. 阶段五：信号拓扑预览与蓝图（Phase 2 → 3）

### 7.1 背景

Phase 1 解决了"能配"的问题，但没有解决"能看懂"的问题。在一个复杂的载具零件中，信号流向可能跨越多个子零件、连接点和子系统。**只读的 SVG 拓扑预览图**让作者一眼看清整个信号网络，而**拖拽连线蓝图**则进一步提供最直观的编辑方式。

### 7.2 Phase 2：只读信号拓扑预览

#### 7.2.1 实现方案

在当前零件内部，新增一个**SVG 信号拓扑预览面板**（位于属性面板下方或独立 Tab）：
- 节点 = 连接点实例 + 子系统实例 + 交互盒 + 特殊目标（`subpart`、`vehicle`）
- 有向边 = `signal_targets`、`control_outputs`、`speed_outputs`、`power_output`
- 布局算法：力导向布局（d3-force 或自实现简易弹簧布局）
- 渲染方式：内联 SVG（无 Canvas，零外部依赖）

### 7.4 技术选型说明

**为什么不直接做 Phase 3？**
- Phase 2（只读预览）**没有交互复杂度**，只需渲染 + 力导向布局，开发周期短（2～3 周）
- Phase 3（拖拽连线）需要完整的 hit-testing、正交折线路由、拖拽交互，开发周期长（4～6 周）
- Phase 2 本身就有独立交付价值：让作者看到信号流向
- Phase 2 的数据模型和渲染层可**直接复用**到 Phase 3

**为什么不用 Canvas/WebGL？**
- Blockbench 生态中所有图编辑都是 DOM + SVG/CSS（动画控制器、Timeline Graph）
- SVG 与 Vue 2 响应式天然兼容，无需处理 Canvas 的 hit-testing
- 信号拓扑的节点数通常不超过 50，SVG 性能完全够用

**为什么不直接用 Blockbench 动画控制器的 DOM 列表实现？**
- 动画控制器的状态机是**线性**排列（CSS Flexbox），无法表达有向图的自由流向
- 动画控制器的连线是 CSS border 拱形（仅水平方向），无法处理正交折线
- 动画控制器没有画布缩放/平移能力
- 详见 [06-技术实现笔记 §16](./06-技术实现笔记.md#16-blockbench-动画控制器与信号拓扑图的差异)
