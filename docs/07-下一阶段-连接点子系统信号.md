> 创建时间：2026-05-18

## 1. 阶段定位与动机

### 1.1 为什么合成一个阶段

原始 [05-开发计划](./05-开发计划.md) 中，阶段三（子系统配置）和阶段四（信号路由表单）是拆开的。但在实际开发中，这三者强耦合：

- **子系统的核心价值**在于生产/消费信号（控制器输出指令、发动机输出转速反馈等）
- **连接点的信号字段**是跨零件信号传递的唯一通道
- 没有子系统标记，信号目标就没有引用对象；没有信号编辑器，子系统就成了"哑组件"

将三者打包为一个阶段，可在交付时让 UGC 作者**真正走通**"座舱发信号 → 连接点转发 → 控制器接收 → 分发到发动机/轮胎"的完整链路。

> **架构注意**：子系统与 Locator **不是**强绑定关系。绝大多数子系统（engine、gearbox、car_controller 等）是子零件下的纯数据条目，只有少数类型（seat、launcher 等）需要可选的 `locator` 或 `connector` 字段。详见 [§1.3](#13-重要架构修正子系统--必须绑定-locator)。

### 1.2 与 05-开发计划 的关系

本方案是原阶段三 + 阶段四的**精简合并版**。比起分别交付，去掉了以下内容以控制工作量：

| 原计划内容 | 处理方式 | 原因 |
|-----------|---------|------|
| 子系统型号动态表单 (3.4) | 推迟到后续 | 型号编辑流程独立于实例编辑，且当前 subsystem_dialog.js 已有骨架 |
| 第2/3批子系统类型专用字段 (3.7/3.8) | 仅保留首批 6 种完整实现 | 其余类型可复用通用 key-value 编辑，不做专属表单 |
| 交互盒 signal_targets (4.11) | 推迟到后续 | 交互盒本身面板尚未实现 |
| 拓扑预览/蓝图 (阶段五) | 完全不纳入 | 独立大工作量 |

### 1.3 重要架构修正：子系统 ≠ 必须绑定 Locator

原计划（和初版方案）设想子系统通过右键 Locator → "标记为子系统" 来创建。但在对 MachineMax 模组实际 JSON 的分析后发现，**绝大多数子系统根本不引用任何 Locator**：

```json
// AE86 底盘 — 8 个子系统中，6 个无 locator
"subsystem.machine_max.car_controller": { "type": "machine_max:car_controller", ... }  // 无 locator
"subsystem.machine_max.engine":        { "type": "machine_max:engine", ... }             // 无 locator
"subsystem.machine_max.gearbox":       { "type": "machine_max:gearbox", ... }            // 无 locator
"subsystem.machine_max.transmission":  { "type": "machine_max:transmission", ... }       // 无 locator
"subsystem.machine_max.left_front_wheel_driver": {
    "type": "machine_max:wheel_driver",
    "connector": "connector.machine_max.left_front_wheel"  // ← 引用连接点，不是 locator
}

// 只有 seat 需要 locator（指定乘坐位置）
"subsystem.machine_max.seat": {
    "type": "machine_max:seat",
    "locator": "left_front_seat_seatpoint"  // ← 唯一需要 locator 的类型
}
```

**结论**：子系统是子零件下的纯数据条目（类似 key-value 映射），创建时不需要也不应该绑定到任何 3D 元素。`locator` 和 `connector` 只是某些子系统类型的**可选字段**，在属性面板中作为普通下拉选择器存在即可。

### 4.2 Track B: 子系统类型注册表与实例面板

#### B.1 抽离子系统类型注册表

将 `subsystem_generator.js` 中的 `getTypeSpecificFields()` 抽离为独立模块 `core/subsystem_types.js`，扩展为带有完整元数据的注册表。每个类型条目包含：

```javascript
// 示例：发动机类型条目
{
    id: 'machine_max:engine',
    displayName: '发动机',
    category: 'power',                // 'power' | 'control' | 'utility' | 'experimental'
    fields: [                          // 类型专属字段列表（原 getTypeSpecificFields 的输出）
        'max_power', 'max_torque', 'idle_rpm', 'idle_torque_ratio',
        'peak_torque_rpm', 'red_line_rpm', 'red_line_torque_ratio',
        'inertia', 'four_stroke', 'cylinder_count', 'drag_coefficients',
        'control_channels', 'sound_map',
    ],
    defaults: {                        // 类型专属字段默认值
        max_power: 100.0, max_torque: 200.0, idle_rpm: 800,
        idle_torque_ratio: 0.15, peak_torque_rpm: 3500,
        red_line_rpm: 7000, red_line_torque_ratio: 0.85,
        inertia: 0.3, four_stroke: true, cylinder_count: 4,
    },
    needsLocator: false,               // 是否需要 locator 字段（如 seat=是, engine=否）
    needsConnector: false,             // 是否需要 connector 字段（如 wheel_driver=是）
    signalOutputs: [                   // 该类型可能的信号输出字段名列表
        'speed_outputs', 'power_output',
    ],
    signalInputs: [],                  // 该类型可能的信号输入字段名列表
}
```

**关键设计——needsLocator / needsConnector**：

| 子系统类型 | needsLocator | needsConnector | 说明 |
|-----------|-------------|---------------|------|
| `engine`, `motor`, `gearbox`, `transmission`, `basic`, `item_storage`, `lighting`, `motor_controller`, `battery`, `signal_convert`, `javascript`, `camera` | `false` | `false` | 纯计算/存储/效果类，无需任何空间引用 |
| `car_controller`, `motorbike_controller` | `false` | `false` | 纯控制逻辑，通过信号路由与其他子系统通信 |
| `seat` | `true` | `false` | 需要 `locator` 指定乘坐位置 |
| `launcher`, `turret`, `fire_controller` | `true` | `true` | 需要 `locator` 指定瞄准/发射点，需要 `connector` 引用控制连接点 |
| `wheel_driver` | `false` | `true` | 通过 `connector` 引用被驱动的车轮连接点 |
| `joint` | `false` | `true` | 通过 `connector` 引用被控制的连接点 |

**API 设计**：

```javascript
// 查询类型元数据
getTypeMeta('machine_max:engine')     // → 完整条目对象
getAllTypes()                          // → 所有 20 个类型的数组
getTypesByCategory('power')           // → power 分类的筛选数组
getTypeSpecificFields('machine_max:engine')  // → ['max_power', ...]（兼容旧接口）
```

**涉及文件**：`src/core/subsystem_types.js`（新建）、`src/generators/subsystem_generator.js`（改为委托新模块）

### 4.3 Track C: 子系统专属字段与信号输出编辑

#### C.1 类型专属字段动态表单

在 `SubsystemPanel.vue.js` 中，根据当前子系统 type 从 `subsystem_types.js` 获取 `fields` 列表，动态渲染表单控件：

- **首批 6 种（完整专属表单）**：`engine`, `wheel_driver`, `seat`, `gearbox`, `car_controller`, `motor`
  - 每个字段带有中文 label、单位说明、step/step 精度
  - 例如 `engine` 的 12 个专属字段（`max_power`, `max_torque`, `idle_rpm`, …）各有一个数值输入框
  - `four_stroke` 用复选框，`sound_map` 用声音资源选择器
- **其余类型**：通用 key-value 编辑器 — 字段名自动补全（从 `fields` 列表中）、值类型自适应（number/string/boolean/object）

#### C.2 信号输出编辑器（动态字段名）

从实际 JSON 分析，子系统信号输出的字段名**随类型变化**，不是固定的三个名字：

| 类型 | 信号输出字段 |
|------|------------|
| `car_controller` | `control_outputs`, `speed_outputs`, `throttle_outputs`, `brake_outputs`, `steering_outputs`, `handbrake_outputs` |
| `engine` | `power_output`（单值）, `speed_outputs` |
| `gearbox` | `power_output`（单值）, `gear_outputs` |
| `transmission` | `power_outputs`（键值对 `{target: ratio}`） |
| `seat` | `move_outputs`, `regular_outputs`, `aim_outputs`, `passenger_num_outputs` |
| `wheel_driver` | 无信号输出字段（仅接收信号 + 引用连接点） |

这些字段名由 `subsystem_types.js` 中每个类型的 `signalOutputs` 数组声明。面板遍历 `signalOutputs`：
- 值为**单字符串**的（如 `power_output`）→ 渲染为下拉选择器（目标：子系统名 / 连接点名）
- 值为**键值对对象的**（如 `speed_outputs: {channel: [targets]}`、`control_outputs: {channel: [targets]}`）→ 渲染为键值对编辑器
- **特殊类型** `power_outputs: {target: ratio}` — transmission 的动力分配表，目标为子系统名，值为减速比（数值输入）

**涉及文件**：`src/ui/SubsystemPanel.vue.js`

#### C.3 子系统面板 → 声音事件编辑

对于支持 `sound_map` 的子系统类型（如 `engine`, `motor`），提供声音事件下拉选择器（从可用声音事件列表补全）。

**涉及文件**：`src/ui/SubsystemPanel.vue.js`、`src/ui/panels/subsystem_panel.html`
