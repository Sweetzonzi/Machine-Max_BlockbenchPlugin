# 04 — 数据模型 (修订版)

> 最后更新：2026-05-14 | 修订：反映 Part-Variant-SubPart 三级架构

---

## 1. 概述

### 1.1 MachineMax 的数据层级

MachineMax 的零件定义是 **Part → Variant → SubPart** 三级结构：

```
parts/wine_fox_hull.json          ← 一个 PartType 注册
├── 顶层字段 (icon, durability_rate, share_durability, ...)
└── variants: Map<String, VariantAttr>
    ├── "default"                 ← 只有一个变体时支持单值简写
    │   ├── model: "machine_max:wine_fox.geo"  ← 模型引用在 Variant 层级
    │   ├── textures, animations, tags
    │   └── sub_parts: Map<String, SubPartAttr>
    │       └── "sub_part.machine_max.main"   ← 只有一个子零件时支持简写
    │           ├── start_bone: "AllHull"     ← SubPart 通过骨骼范围划定
    │           ├── end_bones: [...]
    │           ├── durability, mass, block_collision, ...
    │           ├── hit_boxes: { ... }         ← 碰撞箱
    │           ├── connectors: { ... }        ← 连接点
    │           └── subsystems: { ... }        ← 子系统
    │
    └── "left"                    ← 多变体示例
        ├── model: "machine_max:wine_fox.geo"  ← 同一个模型
        └── sub_parts:
            └── "sub_part.machine_max.main"
                └── start_bone: "LeftFrontDoor"  ← 不同骨骼
```

### 1.2 关键事实

- **一个 `.bbmodel` 模型 → 多个 `parts/*.json` 零件**（wine_fox 有 11 个）
- **模型引用在 Variant 层级**，不在 Part 层级
- **`start_bone`/`end_bones` 在 SubPart 层级**，划定每个子零件使用模型的哪部分骨骼
- **同一零件下不同 Variant 的 `start_bone` 通常不同**（如 door 的左变体取 LeftFrontDoor、右变体取 RightFrontDoor）
- **`connectors/`, `subsystems/`, `materials/` 静态定义跨零件共享**

### 1.3 简写模式（Codec 实现）

由 Java 代码 `PartType.VARIANT_MAP_CODEC` 和 `VariantAttr.SUB_PART_MAP_CODEC` 可知：
- 只有一个变体且名为 `"default"` 时，`variants` 可省略键
- 只有一个子零件且名为 `"sub_part.machine_max.main"` 时，`sub_parts` 可省略键
- 插件生成的 JSON 应遵循相同规则

## 2. 核心数据结构（修订版）

### 2.1 项目配置 — `MMProjectConfig`

一个 `.bbmodel` 文件对应一个 `MMProjectConfig`，包含多个零件定义 + 共享定义。

```javascript
MMProjectConfig = {
    /** 内容包根目录（导出时选择） */
    packDir: string | null,

    /** 项目默认命名空间 */
    namespace: string,                          // "machine_max"

    /** 当前模型文件名（冗余，便于校验） */
    modelFile: string,                          // "wine_fox.bbmodel"

    /** ─── 载具零件列表 ─── */
    parts: {
        [partId: string]: MMPartConfig          // partId = "wine_fox_hull"
    },

    /** ─── 投射物列表 (未来, 待武器系统发布后跟进) ─── */
    projectiles: {
        [projectileId: string]: MMProjectileConfig
    },

    /** 当前前端编辑状态（不持久化） */
    _uiState: {
        activeMode: string,                     // "part" | "projectile" | ...
        activePartId: string,                   // 当前编辑的零件ID
        activeVariantName: string,              // 当前编辑的变体名
    },

    /** ─── 跨类型共享的静态定义 ─── */
    connector_defs: {                           // 导出为 connectors/*.json
        [defId: string]: ConnectorDef           // defId = "machine_max:wheel_mount"
    },

    subsystem_defs: {                           // 导出为 subsystems/*.json
        [defId: string]: SubsystemDef           // defId = "machine_max:engine_v8"
    },

    material_defs: {                            // 导出为 materials/*.json
        [defId: string]: MaterialDef            // defId = "machine_max:structural_steel"
    },

    /** 配置版本号 */
    configVersion: number,
}
```

### 2.2 零件配置 — `MMPartConfig`

```javascript
MMPartConfig = {
    /** ─── 顶层字段（PartType 级）─── */
    icon: string,                               // "machine_max:textures/icon/..."
    vehicle_durability_rate: number,            // 默认 0.8
    vehicle_damage_rate: number,                // 默认 1.0
    vehicle_damage_rate_destroyed: number,      // 默认 0.1
    functional_threshold: number,               // 默认 0.3, 自动 clamp [0,1]
    share_durability: boolean,                  // 默认 true
    max_stack_size: number,                     // 默认 1

    /** ─── 变体列表 ─── */
    variants: {
        [variantName: string]: MMVariantConfig
    },

    /** ─── 元素标记（按变体隔离）─── */
    element_markers: {
        [variantName: string]: {                // "left" / "right" / "default"
            [uuid: string]: {                   // Group/Locator 的 UUID
                type: 'sub_part' | 'hit_box' | 'connector' | 'seat'
                    | 'lighting' | 'subsystem_locator',
                config_ref: string | null,       // 指向 subPart 的键 或 null
            }
        }
    }
}
```

### 2.3 变体配置 — `MMVariantConfig`

```javascript
MMVariantConfig = {
    model: string,                              // "machine_max:wine_fox.geo"
    textures: string | {
        [textureName: string]: string           // 单值时等效 {"default": value}
    },
    animations: string | null,                  // "machine_max:wine_fox" 或 null
    tags: string[],                             // ["machine_max:door", ...]

    /** ─── 子零件列表 ─── */
    sub_parts: {
        [subPartName: string]: MMSubPartConfig
    }
}
```

### 2.4 子零件配置 — `MMSubPartConfig`

```javascript
MMSubPartConfig = {
    /** ─── 骨骼范围（核心）─── */
    start_bone: string,                         // Group 名称，空串表示全部
    end_bones: string[],                        // 排除的 Group 列表

    /** ─── 物理属性 ─── */
    durability: number,                         // 默认 20.0
    mass: number,                               // 默认 25.0 (> 0, 由 Java 校验)
    mass_center: string,                        // Locator 名，默认 "mass_center"
    projected_area: [number, number, number],   // 默认 [0,0,0]

    /** ─── 地形碰撞 ─── */
    block_collision: 'true' | 'ground' | 'false',
    collision_height: number,                   // 默认 -1.0, 台阶辅助高度（仅在 ground 模式生效）
    climb_assist: boolean,                      // 默认 false

    /** ─── 流体动力优先级 ─── */
    hydro_priority: number,                     // 默认 0

    /** ─── 碰撞箱（必需, Java 层抛异常如果为空）─── */
    hit_boxes: {
        [boneName: string]: {
            id: string,                         // 默认 "part"（"part" 时可省略输出）
            type: 'box' | 'sphere' | 'cylinder' | 'capsule' | 'wheel',
            material: string,                   // 资源路径
            thickness: number,                  // 默认 1.0, 单位 mm
            subsystem: string | null,           // 关联子系统全名
            condition: string | null,           // Molang 表达式
            overwrite: object | null,           // { friction: 0.5, ... }
        }
    },

    /** ─── 交互判定区 ─── */
    interact_boxes: {
        [interactName: string]: {
            bone: string,                       // 骨骼名
            signal_targets: object,             // { channel: target[] }
            interact_mode: 'fast' | 'accurate',
            condition: string,                  // 默认 "NOR"
        }
    },

    /** ─── 连接点 ─── */
    connectors: {
        [connectorFullName: string]: {          // "connector.machine_max.left_wheel"
            locator: string,                    // Locator 名称
            definition: string,                 // "namespace:file"
            overwrite: object | null,
            power_target: string | null,
            signal_targets: object,             // { channel: target[] }
            signal_translations: object,        // { fromChannel: toChannel }
            internal: boolean,                  // 默认 false
        }
    },

    /** ─── 子系统 ─── */
    subsystems: {
        [subsystemFullName: string]: {          // "subsystem.machine_max.engine"
            type: string,                       // "machine_max:engine"
            definition: string,                 // "machine_max:engine_v8"
            dynamic_fields: object,             // 如 engine 的 power_output, speed_outputs
        }
    },

    /** ─── 流体动力 ─── */
    hydrodynamics: {
        [boneOrLocatorName: string]: {
            scale: number,
            effective_range: number,
            front_drag: [number, number],
            back_drag: [number, number],
            y_lift: number,
            // ... 各轴向
        }
    } | null,
}
```

### 2.5 连接点静态定义 — `ConnectorDef`

存储在 `MMProjectConfig.connector_defs` 中，跨零件共享，导出为 `connectors/*.json`。

```javascript
ConnectorDef = {
    /** 连接点类型 */
    type: 'Simple' | 'Advanced',            // Simple=公口, Advanced=母口(含关节)

    /** 连接方向 */
    direction: 'xp' | 'xn' | 'yp' | 'yn' | 'zp' | 'zn',

    /** ─── 结构强度 ─── */
    integrity: number,                      // 默认 20.0, 连接点完整度
    damage_reduction: number,               // 默认 2.0, 冲击伤害减免
    damage_multiplier: number,              // 默认 1.5, 冲击伤害倍率
    damage_absorption: number,              // 默认 0.2, (0~1), 冲击能量吸收

    /** ─── 碰撞选项 ─── */
    collide_between: boolean,               // 默认 false, 连接零件间是否发生碰撞

    /** ─── 标签匹配系统 ─── */
    required_tags: string[],                // 必需全部满足的标签
    accepted_tags: string[],                // 至少满足一个的标签
    rejected_tags: string[],                // 全部不满足的标签

    /** ─── 关节属性（仅 Advanced 类型有效）─── */
    joints: JointAttr[],                    // 各自由度的驱动配置
}
```

### 2.6 子系统静态定义 — `SubsystemDef`

存储在 `MMProjectConfig.subsystem_defs` 中，跨零件共享，导出为 `subsystems/*.json`。

字段根据 `type` 极大差异，每种类型有独立的 Schema 定义（见 `schemas/subsystem/static/` 目录）。公共基础字段：

```javascript
SubsystemDef = {
    /** 子系统类型（必填） */
    type: string,                           // 如 "machine_max:engine"

    /** ─── 公共基础 ─── */
    basic_durability: number,               // 默认 20.0, 基础耐久
    pass_damage: boolean,                   // 默认 true, 是否穿透伤害
    limit_damage: boolean,                  // 默认 false, 是否限制伤害上限
    hidden: boolean,                        // 默认 false, 是否隐藏显示

    /** ─── 音效 ─── */
    destroy_sound_event: string | null,      // 摧毁音效
    activate_sound_event: string | null,     // 激活音效
    deactivate_sound_event: string | null,   // 停用音效

    /** 以下为各类型特有字段，此处省略，详见 Schema 文件 */
    // engine: max_power, max_torque, idle_rpm, red_line_rpm, inertia, ...
    // gearbox: forward_gears, reverse_gears, shift_time, ...
    // motor: max_power, max_torque, red_line_rpm, ...
    // seat: mount_offset, view_offset, ...
    // car_controller: steer_speed, steer_return_speed, ...
    // lighting: radius, color, intensity, falloff, ...
    // item_storage: rows, columns, ...
    // wheel_driver: friction, suspension, ...
    // transmission: efficiency, ...
    // basic: (无额外字段)
    // motorbike_controller: lean_angle_max, ...
}
```

### 2.7 材料定义 — `MaterialDef`

存储在 `MMProjectConfig.material_defs` 中，跨零件共享，导出为 `materials/*.json`。

```javascript
MaterialDef = {
    /** ─── 碰撞物理 ─── */
    friction: number,                       // 默认 0.5, 摩擦系数
    restitution: number,                    // 默认 0.3, 弹性系数
    density: number,                        // 默认 1.0, 密度倍率

    /** ─── 装甲等级 ─── */
    armor_thickness: number,                // 等效装甲厚度（mm）, 默认 1.0
    armor_toughness: number,                // 装甲韧性, 默认 0.0

    /** ─── 视觉/音效 ─── */
    hit_sound: string | null,              // 被击中音效
    break_sound: string | null,            // 被破坏音效
    particle: string | null,               // 碰撞粒子效果
}
```

## 3. 元素标记映射 (`element_markers`)

### 3.1 为什么需要按变体隔离

同一个骨骼 `LeftFrontDoor` 在不同零件/变体中有完全不同的角色：

| 零件 | 变体 | start_bone | LeftFrontDoor 的角色 |
|------|------|-----------|---------------------|
| wine_fox_door | left | `LeftFrontDoor` | **子零件起始骨骼** |
| wine_fox_hull | default | `AllHull` | **end_bones 排除列表中的骨骼**（被 hull 排除，因为门是独立零件） |
| wine_fox_door | right | `RightFrontDoor` | 不在骨骼范围内 |

因此 `element_markers` 存储在每个 `MMPartConfig` 内部，按 **变体名 → UUID** 隔离：

```javascript
// 位于 MMPartConfig 内部（见 §2.2）
// 以 wine_fox_door 零件为例：
element_markers = {
    "left": {
        "uuid-of-LeftFrontDoor":    { type: 'sub_part',  config_ref: 'sub_part.machine_max.main' },
        "uuid-of-LeftDoorHitBox":   { type: 'hit_box',   config_ref: 'sub_part.machine_max.main' },
        "uuid-of-left_door_axis":   { type: 'connector', config_ref: 'sub_part.machine_max.main' },
    },
    "right": {
        "uuid-of-RightFrontDoor":   { type: 'sub_part',  config_ref: 'sub_part.machine_max.main' },
        "uuid-of-RightDoorHitBox":  { type: 'hit_box',   config_ref: 'sub_part.machine_max.main' },
        "uuid-of-right_door_axis":  { type: 'connector', config_ref: 'sub_part.machine_max.main' },
    }
}
```

> **设计要点**：`element_markers` 位于 `MMPartConfig`（零件级），而非 `MMProjectConfig`（项目级）。这样切换零件时只需加载当前零件的标记，无需遍历所有零件的标记映射。

### 3.2 UUID 作为稳定引用

Blockbench 模型元素使用 UUID，用户重命名骨骼时 UUID 不变：

```javascript
// Group 对象
group.name  →  用户可改
group.uuid  →  永久不变
```

`element_markers` 的所有键都是 UUID，确保用户重命名骨骼后标记不丢失。

## 4. 数据映射：MMProjectConfig → MachineMax JSON

### 4.1 一个 Part → 一个 `parts/*.json`

```
MMProjectConfig.parts["wine_fox_hull"]  →  parts/wine_fox_hull.json

顶层字段:
  part.icon                              →  "icon"
  part.vehicle_durability_rate           →  "vehicle_durability_rate"
  part.vehicle_damage_rate               →  "vehicle_damage_rate"
  part.vehicle_damage_rate_destroyed     →  "vehicle_damage_rate_destroyed"
  part.functional_threshold              →  "functional_threshold"
  part.share_durability                  →  "share_durability" (默认 true 时省略)
  part.max_stack_size                    →  "max_stack_size" (默认 1 时省略)

变体:
  part.variants[variantName]             →  variants 对象
    variant.model                        →  "model"
    variant.textures                     →  "textures" (单值或键值对)
    variant.animations                   →  "animations" (null 时省略)
    variant.tags                         →  "tags" (空数组省略)
    variant.sub_parts[subPartName]       →  "sub_parts" (单 sub_part 时用简写)
      subPart.start_bone                 →  "start_bone" (空串省略)
      subPart.end_bones                  →  "end_bones" (空数组省略)
      subPart.durability                 →  "durability" (默认 20.0 省略)
      subPart.mass                       →  "mass" (默认 25.0 省略)
      // ... 其余字段同理

多零件:
  MMProjectConfig.parts 中的每一个键    →  生成一个独立的 parts/xxx.json
```

### 4.2 静态定义 → 对应模块文件

```
MMProjectConfig.connector_defs  →  connectors/*.json (每个 defId 一个文件)
MMProjectConfig.subsystem_defs  →  subsystems/*.json (每个 defId 一个文件)
MMProjectConfig.material_defs   →  materials/*.json  (每个 defId 一个文件)
```

### 4.3 简写输出规则

| 条件 | 行为 |
|------|------|
| 只有一个变体且名为 `"default"` | `variants` 直接写对象值, 不包 key |
| 只有一个子零件且名为 `"sub_part.machine_max.main"` | `sub_parts` 直接写对象值, 不包 key |
| 字段值等于默认值 | 不输出（保持 JSON 简洁） |
| 空数组/空对象/null/空串 | 不输出 |

## 5. 持久化方案

### 5.1 主方案：Blockbench Property 机制

在插件 `onload` 时注册 `machine_max_plugin` 为 `ModelProject` 的 Property：

```javascript
// 在 plugin.js 的 onload() 中执行
new Property(ModelProject, 'object', 'machine_max_plugin', {
    default: {},
    exposed: false,
});
```

注册后，Blockbench 的 `.bbmodel` compile/parse 循环会**自动**处理持久化：

```javascript
// 写入 Project.machine_max_plugin 后，Ctrl+S 自动序列化到 .bbmodel：
// ↓ BB 内部 compile() 自动执行
{
    "meta": { ... },
    "name": "wine_fox",
    "elements": [ ... ],
    "outliner": [ ... ],
    "machine_max_plugin": {                 // ← Property.copy() 自动写入
        "$schema_version": 3,
        "parts": { ... },
        "connector_defs": { ... },
        "subsystem_defs": { ... },
        "material_defs": { ... },
        "projectiles": { ... }
    }
}

// 打开文件时 BB 内部 parse() 自动执行：
// Project.machine_max_plugin = { parts: ..., connector_defs: ..., ... }
```

**原理**：Blockbench 的 bbmodel compile（[bbmodel.js:L201-L204](file:///d:/Files/Project_MinecraftMods/blockbench/js/formats/bbmodel.js#L201-L204)）遍历 `ModelProject.properties`，逐个调用 `Property.copy()` 输出。parse 时同理调用 `Property.merge()` 恢复。只有注册过的 Property 才会被纳入循环。

> **注意**：`machine_max_plugin` 内直接存储完整 `MMProjectConfig` 对象。由于 `Property.copy()` 对 object 类型使用 `structuredClone`，深拷贝安全。

### 5.2 备选：独立配置文件

每次保存时额外写入 `.bbmodel.mm_project.json`，作为保底备份：

```json
{
    "$schema_version": 3,
    "bbmodel": "wine_fox.bbmodel",
    "timestamp": 1715678901234,
    "config": { /* MMProjectConfig 的完整序列化 */ }
}
```

### 5.3 加载/保存策略

```
┌─ 加载 ──────────────────────────────────────┐
│ 打开 .bbmodel                                │
│   ↓                                          │
│ BB 自动调用 Property.merge() 恢复            │
│ Project.machine_max_plugin                   │
│   ├── 存在且 version 有效 → ✅ 主方案        │
│   └── 不存在/无效 → 尝试读 .mm_project.json │
│        ├── 存在 → 回填到 Property ✅         │
│        └── 不存在 → 创建空白配置 ✅          │
└──────────────────────────────────────────────┘

┌─ 保存 ──────────────────────────────────────┐
│ Ctrl+S / onUnselect                         │
│   ↓                                          │
│ BB 自动调用 Property.copy() 写入 .bbmodel    │
│   ↓                                          │
│ 插件额外写入 .mm_project.json (保底)         │
└──────────────────────────────────────────────┘
```

### 5.4 拖入导入（反向解析）

用户拖入已有内容包 JSON 文件时，插件从文件路径推断类型并反向解析：

| 文件路径模式 | 解析目标 | 写入位置 |
|-------------|---------|---------|
| `**/parts/*.json` | PartType 结构 | `MMProjectConfig.parts[id]` |
| `**/connectors/*.json` | ConnectorAttr 结构 | `MMProjectConfig.connector_defs[id]` |
| `**/subsystems/*.json` | SubsystemAttr 结构 | `MMProjectConfig.subsystem_defs[id]` |
| `**/projectiles/*.json` | ProjectileType 结构 | `MMProjectConfig.projectiles[id]` |
| `**/materials/*.json` | MaterialAttr 结构 | `MMProjectConfig.material_defs[id]` |

## 6. 配置版本管理

```javascript
const CONFIG_VERSION = 3;   // v1=旧单零件, v2=多零件三级架构, v3=Property持久化 + projectiles支持

const MIGRATIONS = {
    2: function(v1config) {
        // v1 的 part 属性 → v2 的 parts["legacy"] 中
        return v2config;
    },
    3: function(v2config) {
        // v2 的 MMProjectConfig 顶层 version → v3 的内联 $schema_version
        // v2: { version: 2, project: { parts: ... } }
        // v3: { $schema_version: 3, parts: ..., projectiles: {} }
        const v3 = v2config.project || v2config;
        v3.$schema_version = 3;
        v3.projectiles = {};
        return v3;
    }
};
```

    drag_factor: number,                    // 默认 0, 空气阻力系数
    radius: number,                         // 默认 0.05, 碰撞半径 (m)

    /** ─── 弹道参数 ─── */
    base_velocity: number,                  // 参考速度 (m/s)
    base_penetration: number,               // 参考穿深 (mm RHA)
    base_damage: number,                    // 参考伤害值
    base_accuracy_mil: number,              // 基础精度（密位）

    /** ─── 速度-伤害模型系数 ─── */
    penetration_velocity_coefficient: number, // 穿深速度幂指数
    damage_velocity_coefficient: number,      // 伤害速度幂指数
}

## 7. 数据校验规则

### 7.1 结构校验

| 校验项 | 规则 |
|--------|------|
| 至少一个零件 | `parts` 不能为空 |
| 每个零件至少一个变体 | `part.variants` 不能为空 |
| 每个变体有模型 | `variant.model` 不为空 |
| 每个变体至少一个子零件 | `variant.sub_parts` 不能为空 |
| 每个子零件有碰撞箱 | `subPart.hit_boxes` 不能为空（Java 抛异常） |
| 每个子零件质量 > 0 | `subPart.mass > 0`（Java 抛异常） |

### 7.2 引用校验

| 引用字段 | 校验目标 |
|---------|---------|
| `start_bone` | 非空时必须在模型 Group 中 |
| `end_bones[]` | 每个必须在模型 Group 中 |
| `hit_boxes` 键 | 每个必须在模型 Group 中 |
| `locator` (connector/subsystem) | 非空时必须在模型 Locator 中 |
| `mass_center` | 非空时必须在模型 Locator 中 |
| `definition` (connector) | 必须在 `connector_defs` 中 |
| `definition` (subsystem) | 必须在 `subsystem_defs` 中 |

### 7.3 逻辑校验

| 校验项 | 规则 |
|--------|------|
| 零件 ID 不重复 | 同一 namespace 下唯一 |
| 变体名不重复 | 同一零件内唯一 |
| 子零件名不重复 | 同一变体内唯一 |
| 连接点名不重复 | 同一子零件内唯一 |
| 子系统名不重复 | 同一子零件内唯一 |
| 跨零件引用一致性 | 连接点/子系统引用已定义的静态定义 |
