# 子系统机制与信号传输

本教程目标：
- 讲清子系统如何赋予零部件“实际功能”。
- 讲清信号如何在子系统、交互区、连接点、vehicle 之间传递。
- 教你在内容包层（JSON）创建“自己的子系统型号”（复用现有 type）。


---

## 1. 子系统由谁创建

在 `parts/*.json` 的 `sub_parts -> subsystems` 中配置后：
1. 系统读取每个子系统条目（名称 -> 动态属性）。
2. 通过 `type + definition` 找到对应逻辑与静态参数。
3. 运行时创建具体子系统实例并挂到 subpart。

源码锚点：
- `AbstractSubsystemAttr`：动态属性基类，负责 `createSubsystem`。
- `Part.createSubsystems()`：把 JSON 条目实例化并放入 `subPart.subsystems`。
- `AbstractSubsystem`：生命周期、耐久、信号接收与输出。

---

## 2. `type + definition` 的职责分离

- `type`：选择“行为类别”（例如 `machine_max:item_storage`、`machine_max:engine`）。
- `definition`：选择该类别下的“具体型号参数”。

可把它理解为：
- `type` = 类
- `definition` = 配置化实例模板

示例：

```json
"subsystems": {
  "subsystem.machine_max.trunk": {
    "type": "machine_max:item_storage",
    "definition": "machine_max:default_trunk"
  }
}
```

---

## 3. 如何创建“自己的子系统”（内容包层）

这里的“自己的”指：
- 不写 Java 新类型；
- 通过新建 definition 文件，产出你自己的型号。

步骤：
1. 在 `machine_max/subsystems/` 下复制一个接近的官方定义。
2. 改名并调整参数（容量、功率、曲线、耐久、音效等）。
3. 在 part 的 `subsystems` 中用原有 `type` 引用你的新 `definition`。
4. 进游戏验证功能与交互。

高频错误：
- `definition` 路径存在但 `type` 不匹配。
- 名称空间写错导致找不到定义。

---

## 4. 信号通路总览

信号核心接口：
- 发送：`ISignalSender`
- 接收：`ISignalReceiver`
- 存储频道：`SignalChannel`
- 跨连接点转发端口：`SignalPort`

源码锚点：
- `ISignalSender.sendSignalToTarget/sendSignalToAllTargets`
- `ISignalReceiver.getSignalChannel/onSignalUpdated`
- `SignalPort.onSignalUpdated`（跨部件连接后转发）

---

## 5. `signal_targets`：把信号送给谁

`signal_targets` 可出现在：
- 交互区（interact box）
- 连接点（connector）
- 某些子系统动态配置（取决于具体 type）

目标名可指向：
- 某子系统名
- 某连接点名（端口）
- `subpart`
- `vehicle`

示例（交互区把信号发给子系统）：

```json
"interact_boxes": {
  "interact.machine_max.trunk": {
    "bone": "TrunkInteractZone",
    "signal_targets": {
      "trunk_interact": ["subsystem.machine_max.trunk"]
    },
    "interact_mode": "accurate"
  }
}
```

---

## 6. `signal_translations`：跨连接点改频道名

当信号经过连接点端口转发到对侧时，可做频道名映射。

使用场景：
- A 零件内部叫 `interact_left_door`，
- 对侧系统希望接收 `door_toggle`。

可在连接点配置里写：

```json
"connectors": {
  "connector.machine_max.example": {
    "locator": "AttachPoint",
    "definition": "machine_max:fixed_front",
    "signal_translations": {
      "interact_left_door": "door_toggle"
    },
    "signal_targets": {
      "door_toggle": ["subsystem.machine_max.door_ctrl"]
    }
  }
}
```

机制要点：
- 频道转译在 `SignalPort` 层完成。
- 目标频道必须在对侧目标表中存在，否则不会被消费。

---

## 7. 完整最小链路示例

目标：玩家交互 -> 触发信号 -> 子系统响应。

链路：
1. `interact_boxes` 产生 `trunk_interact`。
2. 通过 `signal_targets` 发送到 `subsystem.machine_max.trunk`。
3. 子系统在 `onSignalUpdated` 或内部逻辑中处理。

跨部件时：
1. 本侧 connector 收到频道 A。
2. `signal_translations` 转成频道 B。
3. 对侧 connector 根据 `signal_targets` 把 B 送给目标子系统。

---

## 8. 调试顺序（强烈建议）

1. 先确认子系统本体工作（不跨连接点）。
2. 再加交互区到子系统的直连信号。
3. 最后再引入 connector 转发与频道翻译。

每次只新增一段链路，避免一次性排查过多变量。

---

## 9. 高频问题排查

### 问题 A：子系统完全无反应

检查：
- `type` 与 `definition` 是否匹配。
- 子系统名字是否与 `signal_targets` 目标名完全一致。

### 问题 B：跨部件后信号丢失

检查：
- 连接点是否真的处于连接状态。
- `signal_translations` 映射后的频道名，是否被对侧 `signal_targets` 监听。

### 问题 C：频道名看起来对，但仍不触发

检查：
- 大小写与下划线是否完全一致。
- 是否在错误对象上配置了 `signal_targets`（例如写在了不消费该频道的对象上）。

---

## 10. 能力边界

JSON 层可以完成：
- 子系统型号扩展（新 definition）。
- 信号路由与频道翻译。
- 交互到功能的多数常见玩法。

若你要新增“全新 type 语义”（即全新运行逻辑），需要 Java 扩展（本文不展开）。
