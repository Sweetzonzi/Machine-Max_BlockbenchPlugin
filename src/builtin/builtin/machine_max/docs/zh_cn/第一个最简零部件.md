# 第一个最简零部件（仅碰撞体积）

本教程目标：
- 创建一个“能被加载、能参与碰撞”的最简零部件。

---

## 1. 先理解最小机制

一个零部件（part）在运行时会经历：
1. 读取 `parts/*.json` -> 解析 `variants` 与 `sub_parts`。
2. 每个 `sub_part` 至少要有一个 `hit_boxes` 条目。
3. 根据骨骼上的方块（cube）生成真实碰撞形状。

如果 `hit_boxes` 为空，读取阶段会直接报错。

源码锚点：
- `SubPartAttr`：`hit_boxes` 为空时抛错；按骨骼构建碰撞形状。
- `Part.createSubParts()`：创建 SubPart、命中框与物理体。
- `SubPart`：实际持有刚体并参与碰撞。

---

## 2. 文件放置与命名

建议先复制一份官方简单部件再改名。最简目录示例：

- `machine_max/parts/my_first_part.json`
- 该 JSON 引用的模型与贴图文件（可先复用官方现成资源）

`$schema` 建议保留，便于编辑器提示与校验。

---

## 3. 最小可运行 JSON（无子系统、无连接点）

```json
{
  "$schema": "../docs/zh_cn/schemas/part_definition_schema.json",
  "icon": "machine_max:textures/icon/container_16x16x8_icon.png",
  "variants": {
    "model": "machine_max:container_16x16x8.geo",
    "textures": "machine_max:textures/part/container_16x16x8.png",
    "animations": "machine_max:empty",
    "sub_parts": {
      "sub_part.machine_max.main": {
        "durability": 100.0,
        "mass": 25.0,
        "block_collision": "true",
        "hit_boxes": {
          "Shape": {
            "type": "box",
            "material": "machine_max:structural_steel"
          }
        }
      }
    }
  }
}
```

说明：
- `hit_boxes` 的键（示例里是 `Shape`）必须能在模型骨骼中找到同名骨骼。
- 本示例故意不写 `subsystems`、`connectors`，这是预期行为。
- `type` 可先用 `box`，最稳定，便于初学调试。

---

## 4. 最小成功标准

满足以下 3 点即算通过：
1. 内容包加载无 JSON/Schema 报错。
2. 物体可生成并具有碰撞体积。
3. 没有任何功能性交互（这在本教程是正确结果）。

---

## 5. 高频报错与排查

### 问题 A：提示找不到碰撞骨骼

常见原因：`hit_boxes` 的键名和模型骨骼名不一致。

处理：
- 把键名改成模型里真实存在的骨骼名。
- 先用一个骨骼跑通，再逐步增加。

### 问题 B：提示碰撞形状为空

常见原因：骨骼存在，但骨骼下没有 cube。

处理：
- 在建模软件中确认该骨骼下确实有几何方块。
- 或改为使用有 cube 的骨骼。

### 问题 C：JSON 看起来对但不生效

常见原因：资源路径或命名空间写错。

处理：
- 检查 `machine_max:...` 的路径拼写。
- 确认文件确实在对应目录。

## 6. 给 LLM 的协作提示词模板

可直接给 LLM：

```text
请基于 Machine-Max part_definition_schema，生成一个“仅碰撞、无子系统、无连接点”的最小 part JSON。
要求：
1) 保留 $schema
2) 仅 1 个 sub_part
3) 仅 1 个 box hit_box
4) 不要包含 subsystems/connectors/interact_boxes
5) 输出后再自检字段拼写与路径格式(namespace:path)
```
